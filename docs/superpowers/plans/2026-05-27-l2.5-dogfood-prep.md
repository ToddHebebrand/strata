# L2.5 dogfood prep (valibot) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Verify Strata can ingest and operate on `valibot/library` (811 modules) end-to-end, producing the artifacts (DB, recorded SQL counts, sanity-tool output, commit-gate result) a follow-up keyed session needs to run the paired L2.5 dogfood comparison.

**Architecture:** This is a runbook plan, not a TDD plan. The "implementation" is operating Strata against a new external corpus and recording falsifiable observations. No production code is added; the only new code is two small one-off node scripts that exercise existing `@strata/store` and `@strata/verify` APIs to confirm they work against the corpus. Findings are recorded in a dogfood-results notes file; gaps are logged in `decisions.md` only if reality diverges from the spec.

**Tech Stack:** Existing `@strata/cli` commands (`ingest-batch`, `embed`), direct `@strata/store` + `@strata/verify` API calls in two small TS scripts run via `tsx` or pre-built CLI. `sqlite3` CLI for SQL sanity.

**Spec:** `docs/superpowers/specs/2026-05-27-l2.5-dogfood-prep-design.md`

---

## Conventions

- All paths in this plan are absolute, no shell `~` expansion. Corpus root: `$HOME/code/valibot-l2.5/library`. DB path: `/tmp/valibot.strata.db`.
- "Record output" means: capture the command's stdout/stderr (or relevant excerpt) into the running notes file created in Task 1 (`docs/dogfood-results/l2.5-prep-<UTC-timestamp>.md`). The notes file is the audit trail; do not rely on terminal scrollback.
- "Expected: …" means falsifiable success. Anything else is a finding; record it verbatim and continue if non-fatal, or stop and log a `decisions.md` entry if it blocks the next step.
- `pnpm -r build` may already be current; only re-build when a step explicitly says to.
- Tasks 1–6 are unkeyed. Task 7 is keyed and only runs if the operator opts in.

- [ ] **Step 1: Write the one-off script**

Create `scripts/l2.5-prep-find-declarations.ts`:

```typescript
import { openDb, find_declarations } from "@strata/store";

const dbPath = process.argv[2];
const name = process.argv[3] ?? "parse";
if (!dbPath) {
  console.error("Usage: tsx scripts/l2.5-prep-find-declarations.ts <db> [name]");
  process.exit(2);
}

const db = openDb(dbPath);
const hits = find_declarations(db, name);
console.log(`find_declarations(${JSON.stringify(name)}) -> ${hits.length} hit(s)`);
for (const h of hits.slice(0, 10)) {
  console.log(JSON.stringify(h));
}
if (hits.length > 10) {
  console.log(`(+${hits.length - 10} more, truncated)`);
}
```

## Task 5: Commit-gate sanity — no-op transaction

**Files:**
- Create: `scripts/l2.5-prep-noop-commit.ts` (one-off; deletable after use)
- Modify: `docs/dogfood-results/l2.5-prep-<timestamp>.md` (append gate sanity)

This task confirms that `@strata/verify`'s `commit(db, tx)` — which renders the corpus and runs `tsc --noEmit` on the rendered output — succeeds against valibot/library with zero mutations. If this fails, the corpus isn't usable for the L2.5 dogfood until the underlying tsc issue is resolved, and that's a Strata finding.

- [ ] **Step 1: Write the one-off script**

Create `scripts/l2.5-prep-noop-commit.ts`:

```typescript
import { openDb, begin, rollback } from "@strata/store";
import { commit } from "@strata/verify";

const dbPath = process.argv[2];
const corpusRoot = process.argv[3];
if (!dbPath || !corpusRoot) {
  console.error("Usage: tsx scripts/l2.5-prep-noop-commit.ts <db> <corpusRoot>");
  process.exit(2);
}

const db = openDb(dbPath);
const tx = begin(db, "l2.5-prep-smoke");

try {
  const result = commit(db, tx);
  console.log(JSON.stringify({ ok: result.ok ?? true, diagnostics: result.diagnostics ?? [] }, null, 2));
} catch (err) {
  console.error("commit threw:", (err as Error).message);
  try { rollback(db, tx); } catch {}
  process.exit(1);
}
```

The exact import paths and `commit()` return shape must match the current source — check `packages/verify/src/validate.ts:101` for the signature before running. If the `commit()` signature or return shape has drifted from this script, adjust the script to match (do not adjust the production API to match the script).

- [ ] **Step 2: Build dependencies**

Run:
```bash
pnpm --filter @strata/store build && pnpm --filter @strata/verify build
```

Expected: both complete without error.

- [ ] **Step 3: Run the no-op commit**

Run:
```bash
npx tsx scripts/l2.5-prep-noop-commit.ts /tmp/valibot.strata.db "$HOME/code/valibot-l2.5/library" 2>&1 | tee /tmp/valibot-noop-commit.log
```

Expected: prints a JSON object with `"ok": true` and an empty (or near-empty) diagnostics array.

If `ok: false`: record the diagnostics verbatim. They will reveal whether Strata's `validate()` is calling tsc with the right options for valibot's strict config (`exactOptionalPropertyTypes`, `isolatedDeclarations`, `allowImportingTsExtensions`). Common probable failure modes and their interpretations:

| Symptom | Likely Strata-side cause |
|---|---|
| Errors about `.ts` imports not resolving | `allowImportingTsExtensions` not honored by Strata's tsc config |
| Errors about exact-optional-property mismatch on legal valibot code | `exactOptionalPropertyTypes` not propagated |
| Errors mentioning declaration emit | `isolatedDeclarations` not propagated |
| `@types/*` not found | Same class as the defu typeRoots fix (`decisions.md` 2026-05-26 area) |

In every "fail" case, the finding is a Strata-side gap, not a corpus problem. Stop the runbook, log a `decisions.md` entry naming the specific tsc option / resolution failure, and do not pretend the gate is healthy.

- [ ] **Step 4: Record result**

Append to the notes file under `### Task 5 — commit-gate sanity (no-op tx)`. Include the JSON output and a one-line conclusion. If diagnostics are non-empty, list them in full.

- [ ] **Step 5: Commit**


## Summary

The Phase 1.5 four-task harness shipped and was gated by a cheap N=1
validation round (8 runs, $0.73, `claude-sonnet-4-6`) **before** any N=3
distribution. The gate did its job: it surfaced that the harness is
**invalid, not that the substrate lost** (`decisions.md` 2026-05-15
newest entry). Three distinct, already-diagnosed defects must be fixed
before the harness can produce a number anyone should believe. The
operator approved "fix all 3 properly."

This is a **tight, three-fix remediation spec**. It does not redesign the
benchmark, add tools, change the three Phase 1.5 tools' store logic, or
reopen Phase 1.5 scope. It writes down exactly:

- **Fix R1 — restore the seed `tsc`-clean invariant.** The unmodified seed
  corpus must `tsc --noEmit` cleanly under the scope both configs use,
  because every task's success/quality math assumes a clean baseline.
  `examples/medium/tests/format.test.ts` is written against the
  post-`add_parameter` signature, so the seed is *not* clean under a scope
  that typechecks `tests/`. Resolution: the corpus tsconfig typecheck scope
  is **src-only**; the test-based success signal is `vitest`
  (fail-before / pass-after), not the typecheck.

- **Fix R2 — restore substrate/baseline scorer + quality scope
  equivalence.** Today the baseline `tscNoEmit` typechecks its **whole
  temp tree** (`-p tsconfig.json`, which `include`s `tests/**`) while the
  substrate path only ever typechecks its **rendered changed src
  modules** — two different file sets. `tscClean` (and everything
  downstream of it) is therefore not apples-to-apples, violating the
  BS-Bench-B / BS15-C identical-core integrity property the methodology
  depends on. Resolution: one rule that makes BOTH configs evaluate
  quality **and** the shared per-task criteria over the **identical
  logical scope** (src-only, same tsconfig-derived options, same per-task
  pure core over the same module map), proven by a key-free equivalence
  test.

- **Fix R3 — wall-time / maxTurns budget + the T01/T05 timeout
  diagnosis (BS15-E).** Substrate T01 and T05 hit `error_wall_time` after
  17–22 tool calls. Resolution: make wall-time and maxTurns first-class,
  per-task-overridable runner/session parameters with higher defaults for
  the structurally bigger tasks (quantified from the observed artifact),
  **plus** a diagnosis protocol that classifies each timeout as
  budget-bound vs. BS15-E thrashing vs. genuine tool-ergonomics failure —
  recorded by the operator round, not inferred.

The non-negotiable invariant across all three: **the unmodified seed is
`tsc`-clean under the scope both configs use, and both configs are judged
by one provably-identical core over the identical scope.** If either
cannot be achieved without weakening a task's success criterion or
without a substantial redesign, the relevant bail signal (BS-R1 / BS-R2)
fires and the honest result is "comparison not yet valid," not a rigged
green.

## Background

### The N=1 finding (verbatim-faithful to `decisions.md`)

First keyed 4-task live round, N=1 validation, run as a cheap gate before
the N=3 distribution. It was *not* a clean pattern; diagnosis found the
harness invalid:

- substrate **T03 ✓** and **T08 ✓** (rename + change_return_type drive
  end-to-end via the agent);
- substrate **T01** and **T05** hit `error_wall_time` (17–22 tool calls
  before the wall);
- **baseline 0/1 on ALL four tasks including T03**, which the baseline
  passed cleanly 3/3 in the Phase 4 round.

The baseline regressing to 0/1 on T03 — a task it demonstrably could do
3/3 only one round earlier — is the smoking gun that the harness, not the
baseline, changed.

### The three root causes (faithful to the `decisions.md` newest entry)

**Root cause 1 — seed-clean invariant broken (Pass-3 fixture defect).**
`examples/medium/tests/format.test.ts` is written against the
post-`add_parameter` signature: it asserts a second parameter
(`expectTypeOf(formatTimestamp).parameter(1).toEqualTypeOf<string |
undefined>()`) and calls `formatTimestamp(t, "UTC")`. On the unmodified
seed, `formatTimestamp` takes one argument
(`examples/medium/src/lib/format.ts`: `formatTimestamp(ts: number)`), so a
`tsc --noEmit` that includes `tests/` produces type errors on the seed.
Runtime fail-before / pass-after is the correct shape for a *test-based*
criterion, but it must not make the seed fail typecheck.

**Root cause 2 — substrate/baseline `tsc`-scope asymmetry (scorer
non-equivalence).** The asymmetry lives in
`packages/bench/src/quality.ts` vs. the substrate validate path:

- *Baseline.* `runBaselineTaskTrial` →
  `defaultValidateWorkingTree(treeRoot, …)` → `tscNoEmit(treeRoot)` which
  runs `tsc --noEmit -p <treeRoot>/tsconfig.json`. The copied
  `examples/medium/tsconfig.json` has `"include": ["src/**/*.ts",
  "tests/**/*.ts"]`, so the baseline typecheck **includes
  `tests/format.test.ts`** → fails on the seed-signature mismatch →
  baseline `tscClean:false` → since `validateAfterCommitClean` is part of
  the ten-criterion shared bar, baseline scores **0/1 on every task,
  including T03**.
- *Substrate.* `runAgentForPrompt` computes
  `validateAfterCommitClean` from `validate(db, checkTx)`
  (`packages/verify/src/validate.ts`), which builds `renderedFiles` from
  `listModules(db)` only — and `listModules` reflects what was ingested,
  which is `collectTsFiles(srcRoot)` (src only;
  `packages/agent/src/session.ts` `collectTsFiles` walks
  `corpusRoot/src`). `rootNames` is exactly those rendered src modules.
  The substrate **never typechecks `tests/`.** The substrate
  `resultQuality` is `criteriaQuality(result)` for non-T03 (mirrors
  `validateAfterCommitClean`) or `defaultSubstrateResultQuality` for T03
  (renders only `batch.modules` — src — to a temp dir and `tscNoEmit`s
  that scratch dir, which has no `tests/`).

The two configs are typecheck-judged over **different scopes** (baseline:
src + tests; substrate: src). That is the BS-Bench-B / BS15-C
identical-core violation: "did the task succeed" is no longer the same
question for both configs.

**Root cause 3 — substrate timeouts on T01/T05** (`error_wall_time`,
17–22 tool calls before the wall), independent of scoring. The
`packages/bench/results/phase15-four-task-2026-05-15T20-12-24-426Z.md`
artifact records: substrate **T01** terminated `error_wall_time` at **22
tool calls**; substrate **T05** terminated `error_wall_time` at **17 tool
calls**; substrate **T08** *succeeded* at **18 tool calls / 82,264 ms
wall**; substrate **T03** succeeded at **12 tool calls / 43,880 ms**. The
default wall-time is **240,000 ms** (`runner.ts` `--wall-ms` default) and
maxTurns **25**. T01 and T05 are structurally bigger (add_parameter fans
out to callsites; T05 needs diagnosis-then-fix reasoning). The timeout is
either budget-bound, BS15-E tool-selection thrashing at the 11-tool
surface, or genuine tool-ergonomics failure — each demands a different
response, and the round must *record* which, not infer it. (The strict
11-tool guard in `session.ts` `assertOnlyStrataTools` is the hermetic
contract and is **not** in scope to relax — it is the BS15-E-relevant
surface the agent is selecting tools from, nothing more.)

### What is already proven and must not regress

- substrate **T03** and **T08** paths work end-to-end and must keep
  working (regression guard);
- the Phase 4 genuine T03 regression guards must stay **byte-unchanged**
  (the T03 modules `types/user.ts`, `server/audit.ts`, `users/*` are
  untouched fixtures);
- the existing unit-test suite (152 passing + 2 skipped) must stay green
  key-free;
- `decisions.md` 2026-05-15 "@strata/bench created; T03 scorer core stays
  in @strata/verify" and the per-task scorer-core entries (D12-T01/T05/T08,
  D1) require the **single pure core** discipline — Fix R2 *restores* it,
  it must not fork it.

## Fix R1 — restore the seed `tsc`-clean invariant

### The decision

**Exclude the corpus `tests/` directory from the corpus typecheck
scope.** "The corpus compiles" (`tscClean`) means **`src` compiles**. The
**test-based success signal** for T01/T05 is `vitest`
(`vitestPassed`, fail-before / pass-after) — tests are *run*, not
*type-gated as part of "does the corpus compile."*

### The exact change

`examples/medium/tsconfig.json` — change `include` from:

```json
"include": ["src/**/*.ts", "tests/**/*.ts"]
```

to:

```json
"include": ["src/**/*.ts"]
```

Add an explicit, scoped exclude is unnecessary (dropping `tests/**` from
`include` already removes it from the program); leave `compilerOptions`
otherwise unchanged. This config is the one the baseline copies into its
temp tree (`materializeCorpus` `cpSync`) and the one
`packages/verify/src/validate.ts` `loadCompilerOptions` resolves via
`findNearestTsconfig` for the substrate, so the *same* `include` scope
governs both sides after R2 (see Fix R2).

### Why this keeps every task's success criterion meaningful

- **T01 (`add_parameter`).** Its success is the per-directory
  `"UTC"`/`"local"` callsite wiring plus the new signature, asserted by
  `evaluateT01TextCriteria` (src text) **and** by
  `tests/format.test.ts` *running green* (it `expectTypeOf`s the second
  parameter and calls `formatTimestamp(t, "UTC")`; both only pass once the
  task is done). The type-level assertion inside the test still executes
  under `vitest` (vitest type-checks `expectTypeOf` at run via its own
  type-test path; even taken purely as a runtime smoke, the import + the
  two-arg call only succeed post-task). The signature change is *also*
  independently checked by `evaluateT01TextCriteria`'s
  `signatureHasTimezone` over `src/lib/format.ts`. Nothing about T01's bar
  is weakened: the post-task signature is still required, it is just
  required by the **test** (run) and the **src text core**, not by a
  whole-tree typecheck that mechanically fails the *seed*.
- **T05 (`replace_body` control).** Its success is the half-open
  comparison in `src/lib/dateRange.ts` plus `tests/dateRange.test.ts`
  going green plus the byte-identical-test anti-cheat. The seed
  `dateRange.test.ts` does **not** reference a non-existent signature (it
  calls `isWithinRange(date, start, end)`, which exists on the seed), so
  the seed test only *fails at runtime* (closed-interval bug), which is
  correct fail-before behavior and never a typecheck error. R1 does not
  change T05 at all; it only stops `format.test.ts` poisoning the
  seed-clean check for *all* tasks.
- **T03 / T08.** Neither depends on the `tests/` typecheck. T03's
  criteria are pure src text + the op row; T08's are
  `evaluateT08TextCriteria` over src + callers. Excluding `tests/` from
  the typecheck scope is invisible to them. (Explicit check below.)

### Acceptance check for R1

A key-free check, added to `packages/bench/tests/`:

1. **Seed src is `tsc --noEmit` clean.** Run the corpus typecheck (the
   exact `tscNoEmit` mechanism, post-R1 tsconfig) over the **unmodified**
   `examples/medium` and assert `tscClean === true`. This is the explicit
   acceptance the `decisions.md` entry demands ("the seed `src` alone
   must be verified `tsc --noEmit` clean as an explicit acceptance
   check").
2. **`vitest` still sees and runs the tests.** Assert `vitest.config.ts`
   `include` (`tests/**/*.test.ts`) is unchanged and that
   `hasVitestFiles(treeRoot)` still returns true for the materialized
   corpus, so the test-based signal is not silently disabled by R1. The
   tests directory remains on disk, in the vitest include, and runnable;
   it is only removed from the **typecheck** scope.
3. **T03 path unaffected.** The existing T03 substrate/replay tests and
   `t03.test.ts` stay green unchanged (run as part of `pnpm -r test`),
   confirming the T03 path does not depend on the `tests/` typecheck.

If R1 cannot be done without weakening a task criterion (e.g. the only
way to make the seed clean is to delete the post-signature assertion that
makes T01 meaningful), **BS-R1 fires** — see Bail signals.

## Fix R2 — restore substrate/baseline scorer + quality scope equivalence

This is the methodology-critical fix. A non-equivalent scorer invalidates
every number the harness produces.

### The single rule

**Both configs evaluate `tscClean`, `vitestPassed`, the symmetric
`commitReturnedOk` / `validateAfterCommitClean` pair, AND the per-task
text criteria over the IDENTICAL logical scope: the corpus `src/`
TypeScript modules, under the post-R1 corpus-tsconfig-derived compiler
options, fed through the one pure per-task core in `@strata/verify` over
the same `Map<modulePath, finalText>`.** The only thing that differs
between configs is *where the text comes from* (substrate: render
committed store modules; baseline: read post-edit `src/**/*.ts` off the
temp tree) — never the scope, never the logic, never the tsconfig.

This restores the BS-Bench-B / BS15-C identical-core property exactly as
`decisions.md` D1 / D12-T01/T05/T08 / "@strata/bench created" require: one
pure core; a substrate-render adapter and a baseline-file adapter that
differ only in the `Map` producer.

### Concretely, what changes

**1. `packages/bench/src/quality.ts` — make `tscNoEmit` src-scoped and
identical for both configs.**

`tscNoEmit(treeRoot)` currently runs `tsc --noEmit -p
<treeRoot>/tsconfig.json`. Post-R1 the corpus tsconfig already excludes
`tests/`, so a `-p tsconfig.json` invocation is now src-only **for the
baseline**. To make the substrate's quality `tsc` use the *same* scope and
options, add an explicit src-scoped typecheck helper used by **both**
configs:

- Add `tscNoEmitSrc(treeRoot: string)` that runs `tsc --noEmit -p
  <treeRoot>/tsconfig.json` after asserting that the resolved tsconfig
  `include` is src-only (post-R1). Behaviorally identical to today's
  `tscNoEmit` once R1 lands, but the helper name and an explicit guard
  document the invariant and make a future re-addition of `tests/**` to
  the corpus `include` fail loudly here rather than silently re-introduce
  the asymmetry.
- The baseline path keeps calling this over its **materialized temp
  tree** (`<root>/tsconfig.json` over `<root>/src`).
- The substrate path's `resultQuality` keeps using
  `renderStoreToDir(...)` then `tscNoEmitSrc(outRoot)` — `renderStoreToDir`
  already only writes `batch.modules` (src) and copies the corpus
  `tsconfig.json`, so post-R1 it typechecks the *same* src-only scope
  under the *same* tsconfig. The current
  `defaultSubstrateResultQuality` is **T03-specific** (it re-derives the
  `User`→`Account` rename). Generalize the substrate `resultQuality` so
  every task renders its committed store `src` modules to a scratch dir
  and runs `tscNoEmitSrc` + `vitestRun` there — the **same two probes the
  baseline runs**, over rendered src instead of edited src. The seed
  `tests/` are copied into the scratch tree unchanged (from the corpus)
  so `vitestRun` exercises the identical test files for both configs (the
  test text is never produced by either agent for T01/T05 except T05's
  anti-cheat, which is already handled symmetrically by `T05_TEST_KEY` +
  `seedTestText`).

**2. `packages/bench/src/configs/substrate.ts` — one quality path for all
tasks, mirroring the baseline's two probes.**

Replace the `taskId === "T03" ? defaultSubstrateResultQuality(...)  :
Promise.resolve(criteriaQuality(result))` split with one quality path
that, for **every** task, runs the same two probes the baseline runs
(`tscNoEmitSrc` + `vitestRun`) over the **rendered committed modules the
agent actually produced** — *not* a deterministic re-derivation. The
existing `defaultSubstrateResultQuality` is T03-specific precisely because
it re-derives the `User`→`Account` rename; that re-derivation does not
generalize to T01/T05/T08 (there is no fixed deterministic answer to
re-run — the agent's committed store *is* the output), and re-deriving
would also score a *different* text than the shared core scored,
re-introducing the gap R2 exists to close. Because `runAgentForPrompt`
closes its in-memory DB before returning (the comment in `substrate.ts`
notes this), the scope-equivalent source of the substrate's final src
text is the **`Map<modulePath, finalText>` the per-task
`evaluateT0NCriteria` wrapper already builds internally**
(`renderModule(db, moduleId)` over `batch.modules`, before the DB
closes). The minimal, non-redesign change: have the substrate scorer
wrapper
(`evaluateT01/T05/T08Criteria`, already in `@strata/verify`) **also
return the rendered `Map<modulePath, finalText>`** it builds internally
(it already builds `rendered`), so `substrate.ts` can run `tscNoEmitSrc` +
`vitestRun` against that exact `Map` materialized to a scratch dir — the
identical scope and the identical text the per-task **shared core**
scored. `criteriaQuality` (the `validateAfterCommitClean` mirror) is
removed as the quality source; quality becomes the same render-to-temp +
`tscNoEmitSrc` + `vitestRun` the baseline uses. This keeps the substrate
path's *success* definition (the ten/per-task shared criteria from the
agent run) untouched while making `resultQuality` measured the same way
on the same scope as the baseline.

**3. `packages/bench/src/score.ts` — already correct; assert it.**

`scoreTaskSharedCriteria` / `scoreBaselineTask` already feed one pure core
per task (`evaluateT0NTextCriteria`) over a `Map`. The substrate side
feeds the same cores via `evaluateT0NCriteria` in
`packages/agent/src/session.ts`. **No logic change is needed in
`score.ts`** — the equivalence break was the *quality `tsc` scope*, not
the text-criteria cores. The fix is to (a) make the quality `tsc` scope
identical (above) and (b) prove the text cores are still scope-identical
with a key-free equivalence test (below). The `validateAfterCommitClean`
member of the shared bar must be derived from the **same src-only
scope** on both sides: substrate already is (validate over rendered src
modules only); baseline's `validateAfterCommitClean = probe.tscClean`
becomes the src-only `tscNoEmitSrc` result post-R1/this fix — so the pair
is now symmetric.

### The key-free equivalence acceptance test

Add `packages/bench/tests/scopeEquivalence.test.ts` (key-free, no model):

- Construct one logical post-edit state per task as a
  `Map<modulePath, finalText>` representing a *correct* completion
  (e.g. T01: `format.ts` with the `timezone` param, server callsites
  `"UTC"`, ui callsites `"local"`, HOF callsite untouched).
- Feed it through **the substrate-side path**: the pure
  `evaluateT0NTextCriteria(map)` core (the exact function
  `evaluateT0NCriteria` delegates to).
- Feed the *same map* materialized to a temp `src/` tree + copied
  post-R1 `tsconfig.json` + seed `tests/` through **the baseline-side
  path**: `readModuleMap(srcRoot)` → `scoreTaskSharedCriteria` → the same
  `evaluateT0NTextCriteria` core, plus `tscNoEmitSrc` over that tree.
- **Assert byte-identical criteria booleans** from both paths for: a
  correct completion (all true), a half-done completion (the same subset
  false on both), and a seed/no-op state (the same subset false on both).
- **Assert the `tsc` scope identity directly:** the set of files the
  baseline `tscNoEmitSrc` compiles equals `src/**/*.ts` and excludes
  `tests/**`, and is the same set the substrate scratch-dir
  `tscNoEmitSrc` compiles. A small assertion that
  `hasVitestFiles` is true but the `tsc` program rootNames contain no
  `tests/` path is sufficient and key-free.

If this test cannot be made to pass without per-config special-casing in
the core (e.g. render canonicalization vs. baseline whitespace makes a
regex pass on one and fail the other for a semantically-identical
result), **BS-R2 fires** — that is BS15-C, do not ship the number.

## Fix R3 — wall-time / maxTurns budget + T01/T05 timeout diagnosis (BS15-E)

### (a) First-class, per-task-overridable budget

Today `maxTurns` (default 25) and `wallTimeMs` (default 240,000) are
single global values threaded from `runner.ts` →
`runSubstrateTaskTrial`/`runBaselineTaskTrial` → `runAgentTask` /
baseline `Options`. Make them **per-task overridable** with
structurally-justified higher defaults:

- Add an optional `perTaskBudget?: Partial<Record<BenchTaskId, { maxTurns?:
  number; wallTimeMs?: number }>>` to `RunBenchmarkParams`
  (`packages/bench/src/runner.ts`) and a `--task-budget` CLI form
  (e.g. `--task-budget=T01:maxTurns=40,wallMs=420000`). When a task has no
  override, the global `--max-turns` / `--wall-ms` apply (back-compat;
  T03/T08 unchanged).
- Thread the resolved per-task budget through `BenchTaskRunParams` into
  `runSubstrateTaskTrial` / `runBaselineTaskTrial` (already plumbed as
  `maxTurns`/`wallTimeMs`; the only change is the runner computing the
  effective value per task before the trial loop).
- `runAgentTask` / `runLiveSession` already accept `maxTurns` and
  `wallTimeMs` and apply them to the SDK `Options` and the
  `abortController` timer (`packages/agent/src/session.ts`). **No session
  logic changes** — the budget is already a first-class session
  parameter; R3 only makes the runner *choose it per task* instead of
  globally.

**Starting numbers, derived from the artifact.** Observed substrate
successes: T03 = 12 tool calls / 43,880 ms; T08 = 18 tool calls /
82,264 ms. T01 timed out at 22 calls, T05 at 17 calls, both under the
240,000 ms global wall — meaning the **wall was not the binding
constraint in this round; `maxTurns`/turn-budget pressure or thrashing
was** (22 and 17 calls are below a 25-turn ceiling but the round
terminated `error_wall_time`, so the agent was spending wall on
long-running turns or looping). Set:

| Task | maxTurns default | wallTimeMs default | Justification |
|---|---|---|---|
| T03 | 25 (unchanged) | 240,000 (unchanged) | Proven 12 calls / ~44 s; ample. |
| T08 | 25 (unchanged) | 240,000 (unchanged) | Proven 18 calls / ~82 s; ample, do not perturb a passing path. |
| T01 | **40** | **420,000** | add_parameter fans out to ~8–12 callsites across `src/server`, `src/ui`; each callsite is a reason-then-edit step. 82 s/18 calls was the *largest passing* substrate run; T01 is structurally larger, so a comfortable >2× margin over the T08 success (82 s) is the floor — 420 s and 40 turns give the fan-out room without being open-ended. |
| T05 | **40** | **300,000** | T05 is small in edits (one body flip) but reasoning-dominated: diagnose the failing test, then `replace_body`. 17 calls hit the wall, so the binding constraint is *iterations of diagnose→try→re-check*, not edit volume. 300 s / 40 turns comfortably exceeds the 82 s / 18-call largest success while staying bounded. |

These are **starting** budgets for the *re-validation N=1*, explicitly
quantified from the only data we have (the failing artifact). They are not
a license to inflate until green — see (b) and BS-R3. The cost ceiling
discipline is unchanged: `--trials` still capped at 5; the runner still
prints projected runs; a bigger per-task wall raises the *worst-case*
per-round cost band, so the runner's projected-spend line must include
the per-task budgets so the operator sees the new ceiling before
confirming a round.

### (b) The timeout-classification diagnosis protocol

For the re-validation round, **every** substrate `error_wall_time` or
`error_max_turns` on T01/T05 (or any task) must be classified by the
operator from the session log into exactly one of three buckets, recorded
as a newest-first `decisions.md` entry (the existing per-round session-log
discipline — *recorded, not inferred*):

1. **Budget-bound (made progress, just needed more turns/time).**
   Evidence: the tool-call sequence shows monotonic progress toward the
   task (correct tool selected, transaction open, mutations queued,
   `validate` returning *fewer* diagnostics over time) and the run hit the
   wall mid-productive-work. Response: raise that task's budget by a
   *bounded, logged* increment (e.g. +50%) and re-run **once**. A second
   budget-bound timeout at the raised budget escalates to bucket 3 (it is
   no longer "just needed more time").
2. **BS15-E thrashing (wrong-tool loops / oscillation).** Evidence:
   repeated wrong-tool selection (e.g. `replace_body` where
   `change_return_type` was right; `rename_symbol` on a non-rename;
   begin/rollback oscillation without converging; re-deriving the same
   failed `validate`). Response: this is the BS15-E tool-ergonomics
   finding the Phase 1.5 spec named — surface it; honest prompt iteration
   on tool-selection guidance is in scope (the Phase 3 prompt contract:
   no task-specific recipes), scoring around it is not. Do **not** raise
   the budget to "outlast" thrashing.
3. **Genuine tool-ergonomics failure.** Evidence: the right tool is
   selected but cannot express the task (e.g. `add_parameter` cannot
   resolve a callsite shape — BS15-B territory; `replace_body` rejected
   repeatedly for a body the agent cannot get syntactically valid).
   Response: surface as the substrate limitation it is. This is an honest
   result about the tool surface, reported, not massaged.

The protocol's output is a per-task line in the round's `decisions.md`
entry: `T01: <bucket> — <one-sentence evidence from the session log>`.
The operator round records this; the harness does not auto-classify
(auto-classification would itself be an inference, exactly what the
`decisions.md` discipline forbids).

### The honest-result clause

If, after a **fair, specified** budget (the table above, plus at most one
logged bounded raise per the bucket-1 rule), the substrate genuinely
cannot drive T01 and/or T05, **that is an honest result to report, not
something to massage.** "The substrate struggles on the
reasoning-dominated control / the high-fan-out add_parameter task at the
11-tool surface" is a legitimate, publishable Phase 1.5 finding (it is
exactly the BS15-E open question `strata-design.md` reserves). It is
**not** a budget to keep inflating until the run happens to pass. BS-R3
governs this.

## Out of scope (explicit)

- **No benchmark redesign.** The four-task suite, the
  win/win/control shape, the metrics schema, the distribution math, the
  symmetric retry rule, the report structure — all unchanged.

## Bail signals (carry as exit criteria)

A surfaced wall is more valuable than a papered-over one
(`strata-design.md` § "What success looks like"). Each is logged
newest-first in `decisions.md` in the task that surfaces it.

### BS-R1 — making the seed `tsc`-clean would weaken a task criterion

If the only way to make the unmodified seed `tsc`-clean under the shared
scope is to delete or weaken the assertion that makes a task meaningful
(e.g. removing the `expectTypeOf(formatTimestamp).parameter(1)` /
two-arg-call assertions that make T01's success bar real, rather than
just moving `tests/` out of the *typecheck* scope while keeping it in the
*vitest* scope), **STOP and surface.** The criterion's integrity beats a
green typecheck. The recommended R1 (exclude `tests/` from the typecheck
scope, keep it in vitest) is specifically designed so this does **not**
fire — if implementation finds it does, that is the report.

### BS-R2 — substrate and baseline cannot share one core over the identical scope without a substantial redesign

If the substrate and baseline quality + per-task scorers cannot be made
to evaluate the **identical scope** through the **one** pure core without
a substantial redesign (e.g. render canonicalization vs. baseline
whitespace makes a per-task core regex non-portable, unreconcilable
within the core; or the substrate's committed-store final text genuinely
cannot be produced as the same `Map` the baseline reads), **STOP and
surface.** A non-equivalent scorer invalidates every number. Reporting
"comparison not yet valid" is correct; shipping a rigged green is not.
This is BS15-C / BS-Bench-B; do not fork the scorer to force agreement.

### BS-R3 — T01/T05 still time out at a fair budget due to wrong-tool thrashing

If, at the specified fair budget (the R3 table plus at most one logged
bounded raise for a genuine bucket-1 budget-bound case), substrate T01
and/or T05 still time out **because of bucket-2 wrong-tool thrashing or
bucket-3 tool-ergonomics failure**, that is a real BS15-E finding to
**report honestly**, not a budget to keep inflating until it passes.
Record the bucket and the session-log evidence; the four-task pattern is
then reported with the substrate's honest T01/T05 outcome, not a number
manufactured by an open-ended budget.

## Acceptance — what "remediated" means

Remediation is complete when **all** of the following hold, key-free,
before any operator live round:

1. **Seed src is `tsc --noEmit` clean.** The R1 acceptance check passes:
   the unmodified `examples/medium` typechecks clean under the post-R1
   src-only corpus tsconfig (`tscNoEmitSrc` over the seed → `tscClean ===
   true`), and `tests/` is still present, still in `vitest.config.ts`
   `include`, still runnable (`hasVitestFiles` true).
2. **Substrate == baseline scope, proven by a key-free test.** The
   `scopeEquivalence.test.ts` passes: for each task, the substrate-side
   pure core and the baseline-side
   `readModuleMap`→`scoreTaskSharedCriteria` path return byte-identical
   criteria booleans on the same logical post-edit state, and the
   `tsc` program scope on both sides is exactly `src/**/*.ts` (no
   `tests/`).
3. **Budget params land.** Per-task `maxTurns`/`wallTimeMs` overrides are
   first-class in `runner.ts` and threaded to the session unchanged;
   T03/T08 defaults are byte-unchanged; T01/T05 carry the
   artifact-derived higher defaults; the projected-spend line reflects
   the per-task ceiling.
4. **Existing suite + T03 guards green.** `pnpm -r build` and `pnpm -r
   test` pass (152 + 2 skipped), the genuine T03 regression guards are
   byte-unchanged, and the T03 substrate/replay path is untouched.

**Only then** does the operator re-run the cheap **N=1 validation**
round (4 tasks × 2 configs = 8 runs). If that clears (baseline can do
T03 again; substrate T01/T05 either pass or produce a *classified*
honest finding per the R3 protocol), the operator proceeds to **N=3**.
The N=1→N=3 gate discipline is unchanged from the Phase 4 / `decisions.md`
"validation-before-distribution" record.

## Open questions

Deliberately unresolved on paper; each carries a recommendation.

### 1. Where does the substrate's final src `Map` for `resultQuality` come from?

The substrate agent's in-memory DB is closed before `runAgentForPrompt`
returns, so `resultQuality` cannot re-open it. **Recommendation:** have
the per-task `evaluateT0NCriteria` wrapper in `@strata/verify` (which
already builds the rendered `Map<modulePath,text>` internally) **also
return that `Map`**, and have `substrate.ts` materialize it to a scratch
`src/` tree + copied post-R1 tsconfig + copied seed `tests/` and run the
same `tscNoEmitSrc` + `vitestRun` the baseline runs. This is the minimal
scope-equivalent change and reuses the exact text the shared core scored;
the alternative (re-deriving each task deterministically the way
`defaultSubstrateResultQuality` re-does the T03 rename) does **not**
generalize to the agent's *actual* output for T01/T05/T08 and would
re-introduce a "scored text ≠ quality-checked text" gap. Validate during
implementation that returning the `Map` does not change any existing
`evaluateT0NCriteria` caller's behavior (additive return field only).

### 2. Does `vitest` need to run for the *substrate* quality probe, and is the symlinked `node_modules` reachable from the scratch dir?

The baseline resolves `vitest` via the repo-root `node_modules` symlink
in its temp tree (`decisions.md` D11). The substrate scratch dir built by
`renderStoreToDir` copies only `tsconfig.json`/`package.json`, not a
`node_modules` symlink, so `vitestRun(outRoot)` currently can only pass
*vacuously* (`hasVitestFiles` false because no `tests/` are written
there) for the substrate. **Recommendation:** for scope equivalence the
substrate scratch dir must also receive the seed `tests/` (copied
unchanged from the corpus) **and** the same repo-root `node_modules`
symlink `materializeCorpus` creates, so `vitestRun` is a *real* signal on
both sides over the *same* test files. Validate during implementation
that the symlink + copied seed tests make `vitestRun` reproducibly green
post-task for the substrate exactly as for the baseline; if the substrate
scratch `vitest` cannot be made real, `vitestPassed` is not
scope-equivalent and that is a BS-R2 finding for the *quality* sub-metric
(report it; do not let a vacuous substrate `vitestPassed:true` stand
against a real baseline one).

### 3. Is one bounded budget raise (bucket-1) enough, or will T01/T05 need a different shape of help?

The R3 protocol allows exactly one logged bounded raise for a genuine
budget-bound timeout. **Recommendation:** ship the protocol as specified
and let the re-validation round's classified evidence answer it. If a
*bucket-1* task still times out after the one raise, the rule already
escalates it to bucket-3 (genuine ergonomics) — that is the honest
finding, not a third raise. Do not pre-decide a multi-raise ladder on
paper; that is exactly the "inflate until green" anti-pattern BS-R3
forbids.

## Suggested build order

A suggestion, not a prescription; resequence if a different order
surfaces problems earlier. Steps 1–5 need no API key; only step 6 (the
operator re-validation round) does. `pnpm -r build` and `pnpm -r test`
green at every step boundary.

1. **R1: corpus tsconfig src-only + the seed-clean acceptance check.**
   Edit `examples/medium/tsconfig.json` `include` to `["src/**/*.ts"]`;
   add the key-free seed-`tsc`-clean / tests-still-in-vitest check. This
   alone should flip the baseline back to being *able* to score T03
   (its biggest single symptom). Confirm `pnpm -r test` green.
2. **R2a: `quality.ts` `tscNoEmitSrc` + the scope guard.** Add the
   src-scoped helper with the explicit "tsconfig include is src-only"
   assertion; point the baseline path at it. Unit-test the scope
   assertion.
3. **R2b: generalize the substrate `resultQuality` to the same two
   probes over the same scope** (Open Question 1's recommended `Map`
   return from the verify wrappers + scratch-dir materialize with seed
   `tests/` + `node_modules` symlink). Confirm T03 substrate quality is
   byte-unchanged in outcome.
4. **R2c: the key-free `scopeEquivalence.test.ts`.** The BS-R2 gate — do
   not proceed until substrate and baseline agree byte-identically over
   the identical scope for all four tasks.
5. **R3: per-task budget plumbing.** Add `perTaskBudget` to the runner +
   the `--task-budget` CLI parse + the artifact-derived T01/T05 defaults
   + the projected-spend line update; thread unchanged into the session.
   Unit-test the budget resolution (per-task override beats global;
   T03/T08 unchanged).
6. **Operator re-validation (N=1), then N=3.** Operator runs the

   classification per the protocol, and any BS-R1/R2/R3 observation as a

## Glossary

- **Phase 1.5R.** This remediation: the three already-diagnosed
  harness defects (seed-clean invariant, scorer-scope equivalence,
  budget/timeout diagnosis) fixed so the four-task suite can produce a
  believable number. Not a redesign.
- **Seed-clean invariant.** The unmodified seed corpus `tsc --noEmit`s
  cleanly under the scope both configs use. Broken by the Pass-3
  `format.test.ts` post-signature fixture; restored by R1 (src-only
  typecheck scope; vitest is the test-based signal).
- **Scope equivalence.** Substrate and baseline evaluate quality and the
  per-task shared criteria over the *identical* logical scope (corpus
  `src/`, post-R1 tsconfig options, one pure per-task core over one
  `Map`). The BS-Bench-B / BS15-C identical-core property. Restored by R2.
- **The asymmetry.** Pre-fix, baseline `tscNoEmit` typechecked
  `src + tests` (whole-tree `-p tsconfig.json` with `tests/**` in
  `include`) while the substrate validate/quality path typechecked
  `src` only (rendered ingested src modules). Different file sets →
  non-equivalent `tscClean` → baseline 0/1 on every task incl. T03.
- **Budget-bound vs. BS15-E thrashing vs. tool-ergonomics failure.** The
  three buckets a substrate timeout is classified into by the R3
  protocol, recorded by the operator round, never inferred.
- **BS-R1/R2/R3.** Remediation exit criteria: seed-clean cannot be done
  without weakening a criterion (R1); scope equivalence needs a
  substantial redesign (R2); T01/T05 still thrash at a fair budget (R3).
  Surfacing one is more valuable than working around it.

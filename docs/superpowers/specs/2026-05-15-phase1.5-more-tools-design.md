## Summary

Phase 1.5 of Strata extends the proven `rename_symbol` spine with **exactly
three** new structural mutation tools and adds their benchmark tasks so the
benchmark suite measures a *pattern* instead of a single task:

- **`add_parameter(function_id, name, type, position, default?)`** — add a
  parameter to a function declaration *and* insert a corresponding argument
  at every callsite. Reference-propagation-heavy; the predicted substrate
  win. Enables benchmark **T01**.
- **`change_return_type(function_id, new_type)`** — change a function's
  declared return-type annotation; propagate/repair affected usages as the
  task requires. Reference-aware; predicted win. Enables benchmark **T08**.
- **`replace_body(function_id, new_body)`** — low-level: replace a
  function's body with new text content. Enables the **T05 fix-a-bug
  CONTROL** task, where the hard part is reasoning, not reference-tracking,
  so the substrate is expected to roughly *tie* the file baseline.

This is a phasing extension, not a scope change. The scope is **settled and
not reopened by this spec**: three tools, no more. `extract_function`,
`inline_function`, the class refactor, the type-guard, the error-handling
cascade, and the `create_function`-based add-feature task are explicitly
deferred (named in § "Approach"). This document writes the already-approved
design down rigorously and grounds every part of it in the existing Phase
1/3/4 implementation. The single highest-risk question — *can these three
edits be made on Phase 1's statement-raw-text + identifier-child
representation without lowering work that is bigger than this phase* — is
confronted honestly and per-tool in § "The representation / feasibility
crux", and is the subject of bail signal BS15-A.

## Background

**Phase 0 state.** Phase 0 proved a TypeScript file ingests into
SQLite-backed statement nodes and renders back through `tsc --noEmit`
without semantic loss. Each top-level statement is one node whose payload is
`statement.getFullText(sourceFile)` — *raw source text*. EOF trivia rides
as a sibling `EndOfFileTrivia` node (`decisions.md` 2026-05-14). Render
concatenates children by `childIndex`. Verify is the in-process TypeScript
Compiler API.

**Phase 1 state (`rename_symbol`).** Phase 1 verticalized on one hero
mutation. It added: an `Identifier` child node per `ts.Identifier`
occurrence under a statement, payload `{ text, offset }` where `offset` is
**relative to the parent statement's raw-text payload**
(`packages/ingest/src/identifiers.ts`); a `node_references` index resolved
by the TypeScript `TypeChecker` over a single in-memory `ts.Program`, walked
with the public `getChildren` traversal, public APIs only — *no internal
`.jsDoc`* (`decisions.md` 2026-05-15 BS1); an in-memory transaction overlay
that holds only `identifierMutations: Map<identifierId, { text }>` plus
`pendingOps` (`decisions.md` 2026-05-15 "Transaction overlay stores
identifier text mutations in memory"); an append-only `operations` table;
render-time identifier-text *splicing* into a statement's raw payload by
descending offset (`packages/render/src/splice.ts`); and a separate
`@strata/verify` package owning `validate(db, tx)` and the validating
`commit(db, tx)`. The programmatic T03 path passes all 11 acceptance
criteria. `decisions.md` 2026-05-15 "Phase 1 verticalizes around
`rename_symbol`" explicitly says Phase 1.5 "adds the second structural
mutation and tests whether the same transaction/reference/render spine
generalizes" — *this spec is that test.*

**Phase 3 state (agent drives it).** `@strata/agent` wraps the store/verify
spine as **eight** hermetic in-process `@anthropic-ai/claude-agent-sdk`
tools over one shared `{ db, actor }` context
(`packages/agent/src/tools.ts`): `find_declarations`, `get_references`,
`read_node`, `begin_transaction`, `rename_symbol`, `validate`,
`commit_transaction`, `rollback_transaction`. The hermetic isolation
contract (`tools: []`, `disallowedTools` incl. `LSP`,
`strictMcpConfig: true`, `settingSources: []`) is settled
(`decisions.md` 2026-05-15 "Agent hermetic isolation"). The system prompt
(`packages/agent/src/prompt.ts`) is a static, prompt-cached worldview with
a single worked rename pattern.

**Phase 4 state (the first benchmark).** `@strata/bench` runs the substrate
(`runAgentT03`, reused as-is) and a file-tools baseline N trials on **T03
only**, scored through one provably-equivalent text-criteria core
(`evaluateT03TextCriteria` in `@strata/verify`), reported as distributions.
**The first live round is in (`decisions.md` 2026-05-15 "First T03
benchmark round"):** at N=3, both configs 3/3 success, the substrate used
~3.5× fewer tokens, ~2.2× less wall time, ~3× fewer tool invocations,
~4.9× cheaper — *disjoint* distributions, framed explicitly as an observed
separation at N=3, not a significance claim. That entry's "Revisit when"
names exactly this phase: *"the benchmark broadens past T03 (needs Phase
1.5 tools)."*

**Phase 1 in the design doc.** `strata-design.md` § "Phase 1: Core store
and operations" calls for "5–7 mutation tools." § "Tool set" enumerates
`rename_symbol`, `add_parameter`, `remove_parameter`, `replace_body`,
`create_function`, `delete_node`, `extract_function`, `inline_function`,
`add_import`, `move_declaration`. Phase 1 shipped one. Phase 1.5 takes the
count to **four** (`rename_symbol` + the three here) — squarely inside the
design doc's "5–7" target while honoring the verticalize-then-broaden
discipline that `decisions.md` records for every phase so far.

**What this phase adds, and why a four-task pattern.** One benchmark task
is a demo; the empirical Phase 4 result is a single-task N=3 separation
("a strong directional result, not the final word" — its own words). A
**four-task suite** — T03 rename (done) + T01 add_parameter + T08
change_return + T05 bugfix-control — that shows *big wins on
reference-heavy structural tasks and parity on the reasoning-dominated
control* is a credible, publishable pattern rather than a demo. The
control is the credibility anchor: a benchmark where the substrate wins
everything is suspect; winning the structural tasks and tying the
reasoning task is the defensible claim, and it is the claim
`docs/benchmarks.md` already sets up for T05 ("Substrate may show no win
here", "if Strata shows wins on T01–T04 and parity on T05, that's
evidence the wins are real and substrate-specific, not a measurement
artifact").

## Approach

Extend the proven spine with exactly three mutation tools, chosen as a
**win / win / control** mix, then generalize the benchmark harness so T01,
T05, and T08 plug in alongside T03 with per-task provably-equivalent
scorer cores.

| Tool | Benchmark | Predicted | Why it is in scope |
|---|---|---|---|
| `add_parameter` | T01 | substrate **win** | Reference-propagation-heavy: fan an argument out to every callsite from the reference index. Directly exercises the substrate's central pitch (reference integrity) on a *mutating* operation, where Phase 1 only proved it on text-only rename. |
| `change_return_type` | T08 | substrate **win** | Reference-aware: the compiler-driven cascade through callers; the substrate locates callers deterministically. A *type-position* edit that is not an identifier rename — exercises a different region of the statement than Phase 1. |
| `replace_body` | T05 | **tie** (control) | The hard part is *diagnosis*, not reference-tracking; both configs get the same failing-test signal. The control proves the structural wins are real and not a measurement artifact. |

**Why these three, and why a control (the rationale, written into the
spec as the design record).** The Phase 4 finding showed a decisive
substrate win on T03 at N=3. To move from "a demo" to "a pattern" we need
breadth (more than one task) *and* a falsifiable claim (a task the
substrate should **not** win). `add_parameter` and `change_return_type`
are the two cheapest tasks that are genuinely reference-heavy *mutations*
(T03 only mutated identifier text; these mutate structure and propagate to
callsites/callers), so they test the spine's central thesis on harder
ground. `replace_body` is the cheapest tool that yields a faithful
reasoning-dominated control (T05): a single-function logic fix where the
substrate's reference machinery confers no structural advantage, so the
honest prediction is *parity*. A four-task suite showing **big wins on the
two reference-heavy structural tasks and parity on the reasoning control**
is exactly the credible, publishable shape the project's "what success
looks like" section asks for ("Not necessarily on every task, but on
enough that the architectural argument lands").

**Why not more, why not fewer.** Fewer than these three loses either the
breadth (one structural task is still a demo) or the control (no
falsifiable claim — a suite the substrate sweeps is not credible). More
than these three reopens settled scope and conflates substrate-maturity
risk with measurement risk: each additional tool is a new mechanic, not a
spine reuse. The next-cheapest candidates were weighed and **deferred,
named here so the boundary is explicit**:

- `extract_function` — closure / free-variable / capture analysis; the
  hardest mutation on the list (`docs/benchmarks.md` T02). Out of scope.
- `inline_function` — parameter-binding substitution at the callsite
  (T10). Out of scope.
- the class inheritance→composition refactor (T06) — a large multi-method
  structural rewrite. Out of scope.
- `add_import` + the type-guard application (T09). Out of scope.
- the error-handling `Result<T,E>` cascade (T07). Out of scope.
- the `create_function`-based add-feature task (T04). Out of scope.

These are Phase 2+ work. If a need for one surfaces *while building the
three*, that is a `decisions.md` entry, not an in-scope expansion (the
CLAUDE.md anti-scope-creep discipline; the per-phase verticalization
record in `decisions.md`).

**Every new tool follows the rename spine and respects every invariant.**
Concretely, each of the three:

1. Preserves **stable node IDs across the mutation** — the mutation does
   not change parent/child structure, so `nodeId()` outputs are unchanged
   (`decisions.md` 2026-05-15 "Stable node IDs use path plus structural
   child path"). See § "The representation / feasibility crux" for the one
   place this needs care.
2. Mutates the **in-memory overlay only**; nothing touches canonical rows
   until commit.
3. Requires an **open transaction** (`getOverlay(tx)` throws otherwise,
   exactly as `rename_symbol` does via `queueIdentifierUpdate`).
4. Appends a **single typed operation-log row** (`AddParameter`,
   `ChangeReturnType`, `ReplaceBody`) via `queuePendingOp`, with actor +
   `affected_node_ids_json` + optional reasoning — the canonical history
   invariant.
5. Goes through the **render splice** path
   (`packages/render/src/splice.ts`) — see the crux for the splice-shape
   extension this requires.
6. Is **validatable via `@strata/verify`**: `commit` blocks on non-empty
   diagnostics (the validate-before-commit invariant).
7. Is exposed to the agent as a **hermetic SDK tool over the shared
   `{ db, actor }` context** with a clear English-intent description.
8. Uses **only public TypeScript APIs** — no internal `.jsDoc`-style hacks
   (`decisions.md` 2026-05-15 BS1; the public-`getChildren` discipline).

## The representation / feasibility crux

This is the highest-risk section and the one most likely to reshape the
phase. **Phase 1's representation was shaped by a text-only operation.**
A statement is stored as raw source text. The only addressable sub-parts
are `Identifier` child nodes carrying `{ text, offset }`, where `offset`
is a byte offset *into the parent statement's raw payload*. The overlay
holds only `identifierMutations: Map<identifierId, { text }>`. Render
splices identifier text in by descending offset and asserts `oldText`
matches at the offset before replacing
(`packages/render/src/splice.ts`). `rename_symbol` never had to touch a
*non-identifier* region of a statement.

All three Phase 1.5 tools edit **non-identifier regions** of a statement's
raw text:

- `add_parameter` edits the **parameter list** of a function declaration
  *and inserts an argument expression at each callsite's argument list*.
- `change_return_type` edits (or inserts) the **return-type annotation**
  of a function declaration.
- `replace_body` replaces the **brace-delimited body** of a function
  declaration.

The Phase 1 identifier overlay cannot express any of these. This section
specifies, **per tool**, exactly how the target span is located on the
*current* model, what (if any) lowering/overlay extension is required, and
whether each is a straightforward spine extension or a genuine new
capability. It is honest where a tool needs more than Phase 1 has.

### The shared mechanism: on-demand re-parse + a generalized text-span overlay

The unifying answer (Open Question 1 below; recommended here, validated in
implementation): **do not deepen ingest lowering. Locate spans on demand.**

A function declaration's statement node already holds the *exact raw
source text* of that declaration as its payload (Phase 0:
`statement.getFullText(sourceFile)`). At mutation time, the tool:

1. Loads the statement node's raw payload via `findNodeById` / the
   declaration's statement (the same lookups `rename_symbol` does).
2. Re-parses **just that statement's payload** with
   `ts.createSourceFile(...)` (the public API ingest already uses —
   `packages/ingest/src/index.ts`), with `setParentNodes = true`.
3. Walks the resulting AST with the **public `getChildren` traversal**
   (the BS1-mandated traversal; mirrors `identifiers.ts`) to find the
   precise node whose `.getStart()/.getEnd()` (relative to the re-parsed
   payload, which is exactly the stored payload, so offsets are directly
   the payload offsets) bound the parameter list / return-type annotation /
   body.
4. Produces a **text-span mutation**: `{ statementId, start, end, oldText,
   newText }` against that statement's raw payload, and (for
   `add_parameter`) one analogous span per callsite statement.

This requires **one overlay/splice generalization, scoped into this
phase**, not a lowering extension:

- Phase 1's overlay holds `identifierMutations: Map<id, { text }>`. Phase
  1.5 adds a sibling `textSpanMutations: Map<statementId,
  TextSpanEdit[]>`, where `TextSpanEdit = { start, end, oldText, newText }`
  is a generalization of `IdentifierMutation` (which is the degenerate
  case `start = offset`, `end = offset + oldText.length`). This is a
  **small, bounded** addition to `packages/store/src/transactions.ts`
  (`TxOverlay`), the render path
  (`packages/render/src/index.ts` /`splice.ts` already splices by
  descending offset and already asserts `oldText` — `spliceStatement`
  generalizes from "identifier mutations" to "text-span edits" with the
  same descending-offset, oldText-checked algorithm), and the commit
  materialization in `@strata/verify`
  (`materializeStatementPayloads` already re-renders a statement's payload
  from its mutations and recomputes child identifier offsets by signed
  shift — it extends to also shift identifier offsets by the net length
  delta of text-span edits within the same statement).

**This is the genuine new capability of Phase 1.5, and it is named as such
here, not hand-waved.** Phase 1 could pretend the overlay was
"identifier text only." Phase 1.5 cannot: a function body or parameter
list is not an identifier. The decision is to *generalize the existing
text-splice mechanism* (one overlay map, one splice algorithm already
present in spirit) rather than introduce structured AST-node lowering for
parameters/bodies/return-types. Structured lowering of parameter lists and
bodies is *real* additional work (a parameter node kind, a body subtree,
re-resolution of references inside replaced bodies) and is explicitly
**out of scope** — if a tool turns out to need it, that is **BS15-A
firing**, not an in-flight expansion.

The cost of "raw text spans, not structured nodes": a replaced body's
*interior* references are not re-resolved into `node_references` (Phase 1
already has this exact limitation — `decisions.md` 2026-05-15 "Transaction
overlay" / Phase 1 spec § `get_references` edge cases note re-resolution is
a Phase 1.5+ concern). Phase 1.5 confronts this honestly per tool below
and scopes it: where a tool's correctness depends on the *compiler*
catching a mistake, validate-before-commit is the safety net (the same
design stance Phase 1 took: "the substrate trusts the compiler as the
arbiter of correctness, not its own half-reimplementation of TS scoping
rules").

### Per-tool feasibility

**`add_parameter` — feasible on the current model via on-demand re-parse +
the generalized text-span overlay; the callsite-resolution half is the
real risk.**

- *Declaration edit:* re-parse the function declaration statement's
  payload; the `ts.FunctionDeclaration.parameters` `NodeArray` has a
  precise `.pos/.end` and each element a precise span. Inserting a
  parameter at `position` is a text-span edit at the computed insertion
  offset (end of the parameter before it, or just inside `(` for
  `position = 0`). No lowering needed — the span is found by public-API
  re-parse of text the store already holds verbatim.
- *Callsite edits:* this is where it is **not** a trivial spine
  extension. `rename_symbol` mutated the *declaration's own identifier and
  the identifiers that reference it* — and Phase 1's `node_references`
  index maps reference identifier → declaration identifier. For
  `add_parameter`, a "callsite" is a `CallExpression` whose callee
  resolves to the function — i.e. the *parent expression of a reference
  identifier*. The reference index gives us the reference *identifier*
  node; the callsite is the enclosing call's argument list, located by
  re-parsing the *referring statement's* payload and walking up from the
  identifier's span to the nearest `ts.CallExpression` whose expression is
  that identifier. The argument insertion is then a text-span edit in that
  statement. This is feasible **only if the reference index resolves the
  callsites the way it resolved rename refs** — higher-order uses
  (`.map(formatTimestamp)`) and aliased calls have a reference identifier
  but *no enclosing `CallExpression`*, so they are correctly *not*
  argument-insertion sites but *are* arity-break sites the compiler will
  flag. **BS15-B governs this:** if the reference index does not reliably
  surface all callsites the way it surfaced rename refs, that is a real
  finding about the substrate's central pitch — surface it, do not paper
  it. (`docs/benchmarks.md` T01 deliberately includes a
  `.map(formatTimestamp)` and a template-literal callsite precisely to
  stress this.)
- *Verdict:* spine extension for the declaration edit; **genuine new
  reference-to-callsite resolution work** for the callsite edits — scoped
  into this phase, gated by BS15-B.

**`change_return_type` — feasible on the current model via on-demand
re-parse + the generalized text-span overlay; a clean spine extension.**

- The function declaration statement's payload re-parses to a
  `ts.FunctionDeclaration` with an optional `.type` (the return-type
  annotation node) with a precise span, or, if absent, a precise
  insertion point (after the `)` of the parameter list, before the body
  `{`). Either is a single text-span edit. No lowering needed.
- "Propagate/repair affected usages as the task requires": the reference
  index already locates callers (same mechanism as `add_parameter`'s
  callsite half). The actual per-caller edits the T08 task asks for
  (removing now-redundant `as` casts, rewriting exhaustive `if` chains to
  `switch`) are *agent reasoning per site*, not a mechanical fan-out the
  tool performs — `change_return_type` itself only edits the annotation;
  the agent makes the per-caller structural decisions using
  `rename_symbol`-style exploration + (where a caller body must change)
  `replace_body`. This matches `docs/benchmarks.md` T08's framing
  ("still a reasoning-heavy task per call site"); the tool's *deterministic*
  job is the annotation edit, and that is a clean spine extension.
- *Verdict:* clean spine extension. Lowest-risk of the three.

**`replace_body` — feasible on the current model via on-demand re-parse +
the generalized text-span overlay; a clean spine extension, with one
honest caveat about interior references.**

- The function declaration statement's payload re-parses to a
  `ts.FunctionDeclaration.body` (a `ts.Block`) with a precise span (the
  `{ ... }`). Replacing it is a single text-span edit covering that span.
  No lowering needed.
- *Honest caveat:* the new body's *interior* identifiers are **not**
  lowered into `Identifier` child nodes and **not** resolved into
  `node_references`. This is acceptable for Phase 1.5 because (a) it is
  the *same* limitation Phase 1 already documented and deferred, (b) T05's
  fix is a single-statement comparison flip inside one function with no
  new cross-module references, and (c) validate-before-commit is the
  safety net — a body that references something undefined fails `tsc` and
  `commit` blocks. The tool does **not** attempt to re-ingest the replaced
  body; if a future task needs interior-reference integrity after
  `replace_body`, that is a logged Phase 2 decision, **not** in scope here
  (and would be BS15-A if T05 turned out to need it — it does not).
- *Input-shape note:* Phase 1 stores bodies as raw text, so
  `replace_body` takes **validated body text** (a string the tool wraps
  and re-parses to confirm it is a syntactically valid `ts.Block` before
  queueing the span edit), not a structured AST. This is Open Question 3,
  recommended-and-resolved here; rationale in § "Open questions".
- *Verdict:* clean spine extension; the interior-reference caveat is
  named, scoped, and covered by validate-before-commit.

### Crux summary (the verdict)

All three are feasible on the current statement-raw-text + identifier-child
representation **via on-demand TS re-parse of the statement's own stored
payload + one bounded generalization of the existing text-splice overlay
(`textSpanMutations`, a superset of `identifierMutations`)** — *no ingest
lowering extension*. `change_return_type` and `replace_body` are clean
spine extensions (re-parse one statement, edit one span). `add_parameter`
is a spine extension for the declaration edit but introduces **genuine new
work** in resolving each reference identifier to its enclosing callsite
argument list and editing it — that is the real risk of the phase and is
gated by BS15-B. The one new *capability* (not lowering, but more than
Phase 1 had) is the generalized text-span overlay; it is named here as the
phase's genuine new substrate work and is gated by BS15-A: if any tool
turns out to need *structured* parameter/body/return-type node lowering
rather than text-span edits, **stop and surface** — that reshapes the
phase into something bigger than "three more tools on the rename spine."

## Tool specs

All three follow the `rename_symbol` shape in `packages/store/src/rename.ts`
(read the existing file: declaration lookup → identifier child → reference
walk → queue overlay mutations → `queuePendingOp`). Snake-case export with
a camelCase alias, re-exported from the store barrel
(`packages/store/src/index.ts`), same as `rename_symbol`/`renameSymbol`.
Each returns `void`, mutates the overlay only, does not validate/render/
commit (caller-controlled, exactly like Phase 1).

### `add_parameter`

**Signature.**
`add_parameter(db, tx, functionId, name, type, position, default?) → void`
(store-internal; the agent tool name is `add_parameter` with snake_case
args — see § "Agent integration").

**Semantics.**
1. Look up the function declaration node by `functionId`. If its kind is
   not `FunctionDeclaration` (Phase 1.5 scope: top-level function
   declarations only; methods/arrows are out of scope — log a decision if
   T01's corpus needs more), throw.
2. Validate `name` against the TS identifier pattern (reuse
   `rename.ts`'s `IDENT_PATTERN`). Validate `type` and (if present)
   `default` parse as a TypeScript type node / expression by wrapping and
   `ts.createSourceFile` (public-API check; do not hand-roll a type
   grammar).
3. Re-parse the declaration statement's raw payload; locate the
   `parameters` `NodeArray`; compute the text insertion span for
   `position` (clamped to `[0, params.length]`); produce the parameter
   text `name: type` (or `name: type = default`).
4. Queue a `textSpanMutation` on the declaration statement for the
   inserted parameter.
5. Resolve callsites: for every reference identifier whose `to` is the
   function's declaration identifier (the existing
   `getReferencesByTo` walk `rename_symbol` uses), re-parse the
   *referring statement's* payload, locate the reference identifier's
   span, walk up to the nearest enclosing `ts.CallExpression` whose
   callee is that identifier. For each such callsite, queue a
   `textSpanMutation` inserting the argument at `position` in the call's
   argument list. The argument *value* per callsite is **the caller's
   decision** (the agent supplies it via per-callsite reasoning in the
   T01 task — the per-directory `"UTC"`/`"local"` rule); the tool's
   deterministic guarantee is *every resolvable callsite gets an argument
   slot at the right position*, and any callsite the tool cannot resolve
   to an argument list (HOF/aliased) is left for the compiler to flag (it
   becomes wrong-arity), which is the correct, honest behavior.
6. `queuePendingOp` one `AddParameter` row.

**Operation-log row.** `kind: "AddParameter"`,
`params_json: { function_id, name, type, position, has_default }`,
`affected_node_ids_json: [declarationStatementId, ...callsiteStatementIds]`,
`actor`, optional `reasoning`.

**Edge cases.** Zero callsites → declaration-only edit, still one op row.
`position` out of range → clamp, do not throw (mirrors Phase 1's
"trust the compiler" stance for anything the compiler will catch). A
reference identifier with no enclosing `CallExpression` (a type position,
a HOF reference, an aliased binding) → **not** an argument-insertion site;
recorded in the op row's affected set is the *declaration* edit only for
that path; arity breakage there is surfaced by `validate` (this is the
T01 failure-surface the benchmark deliberately includes). Inserting a
non-defaulted parameter before existing arguments at a callsite that omits
later optionals → compiler flags it; not the tool's job to second-guess.

**Invariants.** Stable IDs (no structural change — text spans only).
Overlay-only. Open-tx required. One op row. Splice path. Validatable.
Public APIs only.

**Agent tool description (the text the model sees — written like an
explanation to another developer).**
> Add a parameter to a function declaration and insert a corresponding
> argument at every callsite the reference graph can resolve. Requires an
> open transaction; mutates the transaction overlay only. You supply the
> parameter name, type, position, and optional default; you are
> responsible for the argument expression passed at each callsite (the
> tool reserves the argument slot — you decide its value per callsite).
> Callsites are found through the reference graph, not text search:
> unrelated identifiers that merely share the name are never touched, and
> a reference used in a position that is not a direct call (passed as a
> value, aliased) is reported via validate as an arity mismatch rather
> than silently mis-edited.

### `change_return_type`

**Signature.** `change_return_type(db, tx, functionId, newType) → void`.

**Semantics.**
1. Look up the function declaration node; if not `FunctionDeclaration`,
   throw.
2. Validate `newType` parses as a TypeScript type node (wrap +
   `ts.createSourceFile`, public-API check).
3. Re-parse the declaration statement's payload; locate the
   `FunctionDeclaration.type` node span if present (replace it), else
   compute the insertion point (immediately after the parameter list's
   `)` and before the body `{`) and produce `: newType`.
4. Queue one `textSpanMutation` on the declaration statement.
5. `queuePendingOp` one `ChangeReturnType` row.

The tool **only** edits the annotation. Caller/usage repair (the T08
`as`-cast removal and `switch`-rewrite work) is agent reasoning using
exploration + `replace_body` per caller, not a fan-out this tool performs
(matches `docs/benchmarks.md` T08).

**Operation-log row.** `kind: "ChangeReturnType"`,
`params_json: { function_id, old_type, new_type }`,
`affected_node_ids_json: [declarationStatementId]`, `actor`, reasoning?.

**Edge cases.** No existing annotation → insert. Identical type → no-op,
queue nothing, append no op row (mirrors `rename_symbol`'s same-name
no-op). Body whose returns are now type-incompatible → surfaced by
`validate`; not repaired by this tool (the agent uses `replace_body`).

**Invariants.** Same eight as above.

**Agent tool description.**
> Change a function declaration's return-type annotation (or add one if
> absent). Requires an open transaction; mutates the overlay only. This
> edits the declared return type only — it does not rewrite the function
> body or its callers. After changing a return type, use validate to see
> which callers or which return statements the compiler now objects to,
> then make those structural changes deliberately (for example with
> replace_body) rather than assuming the change propagated itself.

### `replace_body`

**Signature.** `replace_body(db, tx, functionId, newBody) → void`.
`newBody` is the body **including its braces** (a validated `{ ... }`
text block — see Open Question 3).

**Semantics.**
1. Look up the function declaration node; if not `FunctionDeclaration` (or
   it has no body, e.g. an overload signature), throw.
2. Validate `newBody` parses as a syntactically valid block: wrap it as
   `function __probe__() <newBody>` and `ts.createSourceFile`; reject if
   it does not parse as a `ts.Block`. (Syntactic-only check; *semantic*
   correctness is the compiler's job at `validate`, consistent with the
   Phase 1 stance.)
3. Re-parse the declaration statement's payload; locate the
   `FunctionDeclaration.body` (`ts.Block`) span.
4. Queue one `textSpanMutation` replacing that span with `newBody`.
5. `queuePendingOp` one `ReplaceBody` row.

**Operation-log row.** `kind: "ReplaceBody"`,
`params_json: { function_id, new_body_len }` (the body text itself is
recoverable from the post-commit statement payload; storing only the
length keeps the op row small and avoids duplicating large blobs — log a
decision if replay needs the literal body),
`affected_node_ids_json: [declarationStatementId]`, `actor`, reasoning?.

**Edge cases.** Identical body text → no-op, no op row. A body that
references an out-of-scope symbol → compiles-fails → `commit` blocks (the
safety net; the tool does not pre-check semantics). Interior identifiers
of the new body are **not** lowered/re-resolved (named caveat in the crux;
acceptable for T05; out of scope to fix here).

**Invariants.** Same eight. Note specifically: stable IDs hold because the
statement node ID is unchanged and no child structure changes (the
interior identifier child nodes that *were* lowered for the old body
become stale, but they are never read after a `replace_body` because the
statement payload is now canonical text — this staleness is the
interior-reference caveat, scoped, not a correctness break for T05).

**Agent tool description.**
> Replace a function declaration's entire body with new code you provide
> (include the surrounding braces). Requires an open transaction; mutates
> the overlay only. This is the low-level escape hatch for changes that
> are not a rename, a parameter change, or a return-type change — for
> example fixing a logic bug inside one function. It does not analyze the
> new body's references; rely on validate to confirm the new body
> type-checks before you commit.

## Benchmark generalization

The Phase 4 harness is **T03-specific** by construction
(`runner.ts` `task: "T03"`; `score.ts`/`baseline.ts` hard-wired to
`evaluateT03TextCriteria`; `substrate.ts` calls `runAgentT03`; the bench
`BenchmarkReport.task` is the literal `"T03"`). Phase 1.5 introduces a
**task abstraction** so T01/T05/T08 plug in beside T03 with the *same
substrate-vs-baseline identical-core discipline* the Phase 4 spec
established (BS-Bench-B), without rewriting the proven T03 path.

### The task abstraction

Introduce a `BenchTask` interface (new `packages/bench/src/tasks/`
directory; one module per task) capturing exactly the per-task variation
the current harness has hard-coded:

```
interface BenchTask {
  id: "T01" | "T03" | "T05" | "T08";
  prompt: string;                       // verbatim, imported from one source
  substrate(params): Promise<TrialMetrics>;  // the agent run for this task
  baseline(params): Promise<TrialMetrics>;   // the file-tools run for this task
}
```

- `runner.ts` generalizes from "substrate then baseline" for T03 to "for
  each requested task, for each trial, substrate then baseline";
  `--tasks=T01,T03,T05,T08` selects (default: all four). The trial loop,
  cost projection (BS15 budget discipline / Phase 4 BS-Bench-C), the
  distribution math (`metrics.ts`, untouched — it is already task-
  agnostic), and the symmetric retry counter (`retry.ts`, untouched — it
  is already task-agnostic) are reused as-is.
- `BenchmarkReport.task` becomes `tasks: BenchTaskId[]`; the report writer
  emits **per-task** distributions *and* a **cross-task pattern**
  section (see § "Reporting the pattern + the control").

### Per-task shared scorer cores (the BS15-C / BS-Bench-B discipline)

Each new task gets its own *pure text-criteria core in `@strata/verify`*,
exactly analogous to `evaluateT03TextCriteria` (which already takes
`Map<modulePath, finalSourceText>` and returns booleans, fed by a
substrate render adapter and a baseline file-read adapter — see
`packages/verify/src/t03Criteria.ts` and `packages/bench/src/score.ts`):

- `evaluateT01TextCriteria(modules)` → the T01 success booleans (new
  signature exists in `formatTimestamp`; per-directory `"UTC"`/`"local"`
  wiring present; the `.map` HOF callsite is correct or compiler-flagged;
  no missed template-literal callsite). Regexes/text checks defined once;
  the substrate adapter renders committed store modules to text, the
  baseline adapter reads post-edit `.ts` off the temp tree — *byte-
  identical scoring logic, the scorer cannot tell which produced the
  text*, exactly the T03 invariant.
- `evaluateT05TextCriteria(modules)` → the bugfix booleans: the
  comparison is the half-open form, the failing test now passes, **the
  test file is byte-identical to seed** (the anti-cheat negative — T05's
  "Do not modify the test file" / "byte-identical" success criterion).
- `evaluateT08TextCriteria(modules)` → the return-type-narrowing booleans:
  `getRole`'s annotation is the literal union; no `as string` casts on
  its result remain; callers updated.

Each core lives in `@strata/verify` (not `@strata/bench`) — the **same
acyclicity reasoning the Phase 4 decisions locked**: `@strata/verify`
already owns scoring and is depended on by both `agent` and `bench`;
moving a core to `bench` would reintroduce the `verify/agent/bench` cycle
(`decisions.md` 2026-05-15 "@strata/bench created; T03 scorer core stays
in @strata/verify" and "T03 text-criteria core extracted"). The
substrate-side `evaluateT0NCriteria(db, batch, srcRoot, input)` wrapper
(building the rendered-text `Map` then delegating to the pure core, plus
the symmetric `commitReturnedOk`/`validateAfterCommitClean` pair) mirrors
`evaluateT03Criteria` exactly. The op-row check (`AddParameter` /
`ReplaceBody` / `ChangeReturnType` analog of `operationRowAppended`)
stays a **substrate-only sub-metric**, *not* part of the shared bar — the
identical fairness decision and justification the Phase 4 spec made for
`operationRowAppended` (the baseline has no operation log; a synthetic
file analog either duplicates the text criteria or scores the baseline as
failing every trial for lacking a substrate artifact). BS15-C is the gate:
if any task's substrate and baseline scorers cannot be made provably
identical (e.g. render canonicalization vs. baseline whitespace), that
task's number does not ship — same stop-condition as BS-Bench-B.

### Fixtures: extend `examples/medium` (recommended) vs. per-task trees

**Recommendation: extend `examples/medium` with the T01/T05/T08 target
modules; do not create sibling fixture trees.** Justification, grounded in
the existing setup:

- The Phase 4 spec and `docs/benchmarks.md` Open Questions already settled
  on a **single shared seed** (`examples/medium`) — "Shared-seed is
  cheaper to maintain and more realistic … Recommend shared seed." The
  baseline harness (`materializeCorpus`) recursively copies the *entire*
  `examples/medium` tree and `git init`s it; the substrate harness
  ingests every `.ts` under `examples/medium/src`. Both already treat the
  whole corpus as the world. Per-task trees would fork that proven
  materialization/ingest path three ways for no measurement benefit and
  *reduce* realism (the design doc's "use a real example codebase … not
  toy examples" + CLAUDE.md "Toy examples hide the problems").
- The T01/T05/T08 target modules **do not exist yet** (verified: there is
  no `src/lib/format.ts`, `src/lib/dateRange.ts`, or
  `src/lib/permissions.ts`). They must be **added to
  `examples/medium/src`** as part of this phase, matching
  `docs/benchmarks.md`'s starting-state descriptions: a `src/lib/`
  directory with `format.ts` (`formatTimestamp`, ~8–12 callsites incl. a
  template-literal and a `.map` HOF use, across `src/server/` and a new
  `src/ui/`), `dateRange.ts` (`isWithinRange` with the closed-interval
  bug), `permissions.ts` (`getRole` returning a free-form string).
- T01/T05 require a **test suite the baseline can run** (their success
  criteria are `pnpm vitest run tests/...`). `examples/medium` currently
  has **no own vitest suite** (`decisions.md` 2026-05-15 "Baseline
  temp-checkout"; `quality.ts` `hasVitestFiles` returns
  `vitestPassed: true` when none exist). Phase 1.5 must add a minimal
  `examples/medium/tests/` (`format.test.ts`, `dateRange.test.ts`) **and**
  a `vitest` devDependency / config in `examples/medium/package.json` so
  the *baseline's* `vitestRun` is a real signal, not a vacuous pass. This
  is named here because it is a real fixture-and-deps change with a
  knock-on for the baseline temp-tree materialization (Open Question 2;
  the `decisions.md` "Baseline temp-checkout" entry's "Revisit when: the
  corpus gains its own runtime deps or vitest suite" condition is exactly
  this — Phase 1.5 triggers it and must log the revisit).
- The existing T03 modules (`types/user.ts`, `server/audit.ts`,
  `users/*`) and the `"User"` audit negative are **left untouched** so
  T03 keeps passing unchanged (regression guard: the existing
  `evaluateT03TextCriteria`, `t03.test.ts`, and the Phase 4 substrate
  path must stay green — verified by `pnpm -r test` at every task
  boundary, the Phase 4 plan's "vitest does not typecheck" discipline).

### Reporting the pattern + the control

The report (`packages/bench/src/report.ts`) extends from one task to a
suite. Per task it keeps the existing per-config distribution block
(N/min/max/median/mean/p25/p75/stddev/raw — `metrics.ts` unchanged) and
the overlap/“no separable signal” honesty note. It adds a **cross-task
pattern** section that states the claim and its falsifier explicitly:

> Structural/reference-heavy tasks (T01 add_parameter, T03 rename, T08
> change_return): \<separation summary per metric\>. Reasoning-dominated
> control (T05 fix-a-bug): \<overlap/parity summary\>. The pattern holds
> iff the structural tasks separate **and** the control does not;
> overlapping structural tasks or a separated control are reported as
> stated, not massaged.

BS15-D governs control integrity (below). The "report distributions, not
means; overlap is a legitimate result; raising N is an explicit budgeted
operator decision" discipline from Phase 4 (BS-Bench-D) carries over
verbatim and applies per-task.

## Agent integration

The agent gains three tools, going from **8 to 11**. Registration is
mechanical and mirrors the existing eight in
`packages/agent/src/tools.ts`: a `tool(name, description, zodShape,
handler)` over the shared `{ db, actor }` context, added to the
`createStrataTools` array and to `STRATA_TOOL_NAMES` (which feeds
`STRATA_QUALIFIED_TOOL_NAMES`, the hermetic allow-list, and the runtime
`assertOnlyStrataTools` invariant guard — all three new names must be in
that list or the hermetic guard will (correctly) reject the session;
this is a registration checklist item, not new mechanism). Zod arg
shapes reuse `nodeIdSchema`/`txHandleSchema`; `add_parameter` adds
`name`/`type`/`position`/`default?`, `change_return_type` adds
`new_type`, `replace_body` adds `new_body`.

**Prompt updates.** `STRATA_SYSTEM_PROMPT` is a static, prompt-cached
worldview with a single worked rename pattern. Phase 1.5 makes **minimal,
additive** changes: extend "The tool surface" section with one
plain-English sentence per new tool (matching the existing per-tool
sentences' register), and add a short "choosing the right mutation"
paragraph (rename for symbol names; add_parameter for new parameters +
callsite fan-out; change_return_type for the declared return type;
replace_body as the low-level escape hatch for body logic changes — and
that the structural tools are preferred over replace_body when the change
*is* a rename/param/return change, so the operation log records intent).
The single worked pattern stays rename (it is the archetype); the prompt
must **not** encode task-specific recipes (the Phase 3 prompt contract:
"No benchmark-specific identifiers, no scripted recipe").

**BS15-E governs the surface growth.** Going from 8 to 11 tools may
degrade the model's tool selection (picking `replace_body` when
`change_return_type` is the right structural op, or missing
`add_parameter`'s callsite fan-out). Prompt iteration to fix tool-
selection confusion is **in scope**; faking it (e.g. task-specific prompt
recipes, or scoring a task as a substrate success when the agent reached
it via the wrong tool) is not. If the agent cannot reliably pick the
right mutation tool per task after honest prompt iteration, that is a
finding to surface (the tool-granularity open question in
`strata-design.md` § "Open design questions" #3, observed empirically),
not papered over.

## Bail signals

Exit criteria for "stop and surface, don't work around" — Phase-1-spec
rigor. A surfaced wall is more valuable than a papered-over one
(`strata-design.md` § "What success looks like"). Each is logged
newest-first in `decisions.md` in the task that surfaces it (the
per-task-decisions convention from the Phase 1 plan's Amendment B).

### BS15-A — feasibility: a tool needs deeper lowering than is scoped

If `add_parameter`, `change_return_type`, or `replace_body` **cannot** be
implemented as on-demand-re-parse + text-span overlay edits on the current
representation — i.e. it genuinely requires *structured* node lowering of
parameter lists / bodies / return-type annotations (a parameter node kind,
a body subtree, interior-reference re-resolution) to be correct for its
benchmark task — **STOP and surface.** That reshapes Phase 1.5 from "three
more tools on the rename spine" into a lowering project; do not hack a
half-lowering in flight. (The crux argues this does *not* fire for any of
the three; BS15-A is the check that the argument held in implementation.)

### BS15-B — callsite propagation for `add_parameter`

If the `node_references` index does **not** reliably resolve all callsites
the way it resolved rename references — e.g. higher-order
(`.map(formatTimestamp)`) or aliased calls have a reference identifier but
the tool cannot correctly classify "is this an argument-insertion site or
a compiler-flagged arity break" — that is a **real finding about the
substrate's central pitch (reference integrity)**, not an implementation
nuisance. Surface it (which callsite shapes resolve, which do not, and
whether the miss is a substrate limitation or an implementation defect, as
BS1 was investigated before being accepted/rejected). Do not paper over a
missed callsite by falling back to text search — that would silently
abandon the substrate's whole argument.

### BS15-C — scorer equivalence per new task

Each new task's substrate scorer and baseline scorer **must** share one
provably-identical pure text core (the T03 `evaluateT03TextCriteria` /
BS-Bench-B discipline), fed by a substrate-render adapter and a
baseline-file adapter, scoring "final text of each module" with byte-
identical logic. If a task's two scorers cannot be made provably
equivalent (e.g. render canonicalization vs. baseline whitespace makes a
regex pass on one and fail the other for a semantically-identical result,
unreconcilable within the shared core), **that task's comparison is
invalid — do not ship its number.** Surface the specific divergence and
either canonical-form both sides before scoring (legitimate) or log that
task's scorer as not config-portable.

### BS15-D — control integrity (T05 must be a fair parity task)

If T05 is accidentally designed so the **substrate has a structural
advantage** (the bug fix happens to be expressible as a rename/param/
return-type structural op the baseline must do by hand) **or the baseline
is handicapped** (denied the failing-test signal, given a worse prompt,
run on a tree where `vitest` cannot actually run), the control is
worthless and the whole "wins are real, not artifact" claim collapses.
The spec's position (see below) is that T05 is fair *iff*: the fix is a
single-statement logic change inside one function body
(`isWithinRange`'s `<=` → `<`), expressible by **both** configs as a
localized edit with **no** reference fan-out advantage to the substrate
(`replace_body` is the substrate's tool here — a low-level body swap that
confers none of the reference-graph leverage that wins T01/T03/T08); both
configs get the **same verbatim prompt and the same failing
`tests/dateRange.test.ts` signal**; the test file is byte-identical-to-
seed checked **identically** for both (anti-cheat symmetric); and the
baseline's `vitest` actually runs in its temp tree (the Open Question 2
deps work — if `vitest` cannot run for the baseline, T05 is not fair and
this fires). If any of those does not hold, **stop and surface**; do not
tune T05 to manufacture parity *or* a substrate win.

### BS15-E — agent ergonomics at the larger tool surface

If, after honest prompt iteration, the agent at **11 tools** cannot
reliably select the correct mutation tool per task (e.g. uses
`replace_body` to do what `change_return_type` should, defeating the
operation-log-as-intent invariant; or misses `add_parameter`'s callsite
fan-out and hand-edits via the wrong tool), **surface it** as the
empirical answer to `strata-design.md`'s tool-granularity open question.
Prompt iteration is in scope; scoring around the confusion (counting a
wrong-tool path as a substrate success, or adding task-specific prompt
recipes that violate the Phase 3 prompt contract) is not.

## Open questions

Deliberately unresolved on paper; answered by what implementation
observes. Each carries a recommendation to validate.

### 1. On-demand re-parse vs. an ingest lowering extension

**Recommendation: on-demand re-parse of the statement's own stored raw
payload (`ts.createSourceFile` + public `getChildren` walk) + the
generalized `textSpanMutations` overlay; no ingest lowering extension.**
Rationale: the statement payload *is* the verbatim source
(`statement.getFullText`), so a re-parse is exact and needs no schema
change; the splice mechanism already exists and already does descending-
offset, oldText-checked edits; structured lowering of params/bodies/
return-types is real Phase 2 work and unnecessary for these three tasks
(the crux argues this per tool). Validate during implementation: confirm
re-parsed spans map exactly onto the stored payload for all three target
modules; if `add_parameter`'s callsite walk needs more than this (BS15-B),
or any tool needs structured nodes (BS15-A), log it as a newest-first
decision rather than silently lowering.

### 2. Fixtures: extend `examples/medium` vs. per-task trees, and the baseline-deps knock-on

**Recommendation: extend `examples/medium` (single shared seed) — add the
`src/lib/{format,dateRange,permissions}.ts` + `src/ui/` callsite modules +
a minimal `tests/` suite + a `vitest` dev dep/config in
`examples/medium/package.json`; do not create per-task trees.** Rationale:
the Phase 4 spec and `docs/benchmarks.md` already settled shared-seed; the
substrate-ingest and baseline-`materializeCorpus` paths already treat the
whole corpus as the world; per-task trees would triple-fork that proven
path for no benefit. **The real open part is the baseline-deps knock-on:**
T01/T05 success criteria are `pnpm vitest run`, and `examples/medium`
currently has no vitest suite/deps, so `materializeCorpus`
(recursive-copy, no install — `decisions.md` 2026-05-15 "Baseline temp-
checkout") will *not* have a runnable `vitest` in the temp tree. Validate
during implementation: either add `vitest` to `examples/medium` and have
`materializeCorpus` pre-install once into the temp tree, or symlink the
resolved dep — whichever makes `pnpm vitest run` reproducibly green for
the *baseline*. This is exactly the "Revisit when: the corpus gains its
own vitest suite" condition that decision named; log the resolution as a
newest-first entry. (If it cannot be made green for the baseline, T05/T01
fairness fails — BS15-D for T05.)

### 3. `replace_body` input shape (structured AST vs. validated text)

**Recommendation: validated body text** (a `{ ... }` string the tool
syntactically validates by wrapping + `ts.createSourceFile` before
queueing the span edit), **not** a structured AST input. Rationale: Phase
1 stores bodies as raw text and renders canonically; a structured-AST
input would require a body-construction API and a structured-render path
that does not exist and is out of scope (it is the deferred Phase 2
lowering). Validated text matches the storage model, keeps the tool a
clean spine extension, and leans on validate-before-commit for semantic
correctness (the consistent Phase 1 stance). Validate during
implementation that the syntactic pre-check rejects obviously-broken
bodies early (better agent ergonomics) without attempting semantic
analysis the compiler will do at `validate`. If T05's agent struggles to
produce a valid body string (BS15-E), log it; do not switch to a
structured input mid-phase (that is a re-scope, not a tweak).

A suggestion, not a prescription; resequence if a different order
surfaces problems earlier. `pnpm -r build` and `pnpm -r test` green at

round) does.

1. **Generalize the overlay + splice (the crux mechanism).** Add
   `TextSpanEdit` / `textSpanMutations` to the store overlay; generalize

2. **`spanReparse.ts`.** The shared public-API span locator

6. **Fixtures.** Add `src/lib/{format,dateRange,permissions}.ts`,

## Glossary

- **Phase 1.5.** This phase: three new mutation tools (`add_parameter`,
  `change_return_type`, `replace_body`) on the proven `rename_symbol`
  spine, plus the benchmark generalization that turns the suite from one
  task into a four-task pattern.
- **The spine.** The Phase 1 path every mutation follows: declaration
  lookup → (Phase 1) reference walk → queue overlay mutations →
  `queuePendingOp` → render splice → `@strata/verify` validate-before-
  commit (`packages/store/src/rename.ts` is the archetype).
- **Text-span edit (`TextSpanEdit`).** The Phase 1.5 overlay primitive:
  `{ statementId, start, end, oldText, newText }` against a statement's
  raw payload. The Phase 1 identifier mutation is its degenerate case
  (`start = offset`, `end = offset + oldText.length`). The phase's one
  genuine new substrate capability; gated by BS15-A.
- **On-demand re-parse.** Locating a parameter list / return-type
  annotation / body span at mutation time by `ts.createSourceFile` over
  the statement's *own stored raw payload* + a public `getChildren` walk
  — no ingest lowering, no internal TS APIs (BS1 discipline).
- **Callsite (for `add_parameter`).** A `ts.CallExpression` whose callee
  is a reference identifier resolving (via `node_references`) to the
  function's declaration identifier. Reference identifiers with no
  enclosing call (HOF/aliased/type positions) are *not* argument-
  insertion sites; resulting arity breakage is surfaced by `validate`,
  not silently mis-edited. BS15-B governs whether the index resolves
  these reliably.
- **Control (T05).** The reasoning-dominated bug-fix task where the
  substrate is predicted to **tie** the file baseline (no reference-graph
  advantage). The credibility anchor: the four-task claim is "wins on
  structural tasks **and** parity on the control." BS15-D governs its
  fairness.
- **Per-task shared scorer core.** A pure
  `Map<modulePath, finalText> → booleans` function in `@strata/verify`
  (one per task, modeled on `evaluateT03TextCriteria`), fed by a
  substrate-render adapter and a baseline-file adapter so "did the task
  succeed" is byte-identical for both configs. BS15-C is the gate.
- **Substrate-only sub-metric.** The per-task op-row check
  (`AddParameter`/`ChangeReturnType`/`ReplaceBody` analog of T03's
  `operationRowAppended`): recorded for substrate trials, `null` for
  baseline, **not** part of the shared pass/fail bar — the same fairness
  decision the Phase 4 spec made for `operationRowAppended`.
- **Bail signal (BS15-A..E).** An exit criterion: feasibility (A),
  callsite reference integrity (B), per-task scorer equivalence (C),
  control integrity (D), agent ergonomics at 11 tools (E). Surfacing one
  is more valuable than working around it.

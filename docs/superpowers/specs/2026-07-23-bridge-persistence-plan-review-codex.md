## Blocker

1. **The unchanged Gate 3 harness cannot produce the required PASS.** Task 11 requires the existing harness to exit 0/PASS (`docs/superpowers/plans/2026-07-23-bridge-persistence-slice.md:282-294`), but the medium-memory placeholder is always `INCONCLUSIVE` (`packages/live-compare/src/gate3/run-big.ts:197-217`, `packages/live-compare/src/gate3/run-big.ts:403-406`). Medium corpus state includes that placeholder (`packages/live-compare/src/gate3/report.ts:120-138`), overall state includes medium (`packages/live-compare/src/gate3/report.ts:159-172`), and `INCONCLUSIVE` exits 1 (`packages/live-compare/src/gate3/report.ts:208-227`). Task 9’s replacement true-process memory guard is not wired into Task 11’s verdict (`docs/superpowers/plans/2026-07-23-bridge-persistence-slice.md:246-260`). This needs an operator-approved A3 amendment: retain the existing wall-time schedules, seeds, windows, bootstrap method, and thresholds, but replace the obsolete memory component with Task 9’s true-process predicate or define the exit decision as wall/lifecycle plus that predicate. Without A3, the stated exit gate is internally impossible.

2. **The proposed persistent mirror can hydrate an unpublished speculative generation.** Publication constructs `next` and invokes readiness against it before final authority checks and durable commit (`crates/strata-coordination/src/coordination/publication.rs:522-578`, `crates/strata-coordination/src/coordination/publication.rs:697-783`). The plan says missing delta coverage causes a full rehydrate to the exact target (`docs/superpowers/plans/2026-07-23-bridge-persistence-slice.md:194-198`). If readiness analyzes speculative `next`, no committed delta exists yet, so the fallback can advance the persistent mirror to a graph that is subsequently invalidated. Because synchronization is forward-only, the canonical generation can then be behind the mirror. Persistent synchronization must be restricted to published canonical snapshots; speculative readiness must use an isolated one-shot mirror. Committed deltas may enter the log and advance the persistent mirror only after `PublishOutcome::Published` and the in-memory publication (`crates/strata-coordination/src/coordination/publication.rs:803-829`). Add a failpoint test in which speculative readiness completes but the final check invalidates publication, proving the persistent mirror remains at the prior generation.

3. **Attestation, synchronization, and semantic dispatch are not atomic under daemon concurrency.** The service spawns one thread per connection (`crates/strata-coordination/src/service/server.rs:49-55`), while the proposed host exposes separate `last_attestation`, `sync`, and `request` operations (`docs/superpowers/plans/2026-07-23-bridge-persistence-slice.md:128-134`, `docs/superpowers/plans/2026-07-23-bridge-persistence-slice.md:193-198`). Another service thread can synchronize the worker between the first thread’s attestation check and semantic request. Replace these with a single `request_at(graph_identity, frame)` operation holding one host mutex across attestation validation, delta selection or hydration, synchronization-response verification, and semantic request/response. Queueing for that mutex must consume the request deadline. Add orchestrated G/G+1 concurrency tests. Since the design is serialized and single-flight, “out-of-order responses matched by requestId” (`docs/superpowers/plans/2026-07-23-bridge-persistence-slice.md:136`) should be removed; a mismatched or unsolicited response should poison and restart the worker.

## Major

1. **The canonical digest encoding is not portable as specified.** The plan represents generation as a JavaScript/Rust `number` and assumes `serde_json::to_string` and `JSON.stringify` produce identical canonical bytes (`docs/superpowers/plans/2026-07-23-bridge-persistence-slice.md:69-75`, `docs/superpowers/plans/2026-07-23-bridge-persistence-slice.md:109-115`). Existing bridge protocol deliberately represents Rust `u64` values as canonical decimal strings (`crates/strata-coordination/src/bridge/protocol.rs:13-75`, `packages/kernel-bridge/src/protocol.ts:15-22`). Retain a dual state digest, but encode generation as a decimal string and define an explicit canonical writer or top-level array rather than relying on object-key behavior. Specify the string comparator and add fixtures for maximum `u64`, maximum safe child index, escaping, non-BMP characters, and U+2028/U+2029, plus randomized cross-language differential tests. A delta-chain digest alone would attest receipt and ordering, not the resulting SQLite state; a faulty mirror could advance the chain while remaining silently divergent.

2. **The delta log needs count and byte bounds.** The plan retains deltas since service start (`docs/superpowers/plans/2026-07-23-bridge-persistence-slice.md:194`), while the service loop is unbounded (`crates/strata-coordination/src/service/server.rs:49-59`). Use a `VecDeque` capped by both entry count and serialized size—for example, 4,096 deltas or 16 MiB, whichever is reached first. If the requested base predates the retained floor or the batch cannot fit one request frame, full-hydrate the exact target and attest it. Extract an owned or reference-counted batch while holding the log lock so concurrent eviction cannot affect an in-flight sync. Test exact boundaries, too-old generations, and append-during-sync truncation. Preserve the current asymmetric frame limits—32 MiB request and 16 MiB response (`packages/kernel-bridge/src/worker.ts:15-18`, `crates/strata-coordination/src/bridge/process.rs:48-54`)—rather than applying the request limit to every frame.

3. **The savepoint design gives logical isolation, but the proposed oracle is incomplete.** The worker uses an in-memory SQLite database (`packages/kernel-bridge/src/snapshot.ts:75-101`), so raw database-file identity, WAL contents, and vacuum layout are not relevant. However, candidate execution creates transaction rows and successful commits write operation rows and generated metadata (`packages/store/src/transactions.ts:59-68`, `packages/store/src/transactions.ts:253-277`). The proposed `(generation, graph digest)` postcheck covers nodes and references but not transactions, operations, or auxiliary tables (`docs/superpowers/plans/2026-07-23-bridge-persistence-slice.md:216-220`). Assert a canonical logical database fingerprint covering all relevant tables before and after every candidate, plus integrity and foreign-key checks and the external `MirrorState`. Test successful candidates, validation failure, thrown exceptions, identity errors, and JavaScript overlay cleanup.

4. **The performance projection is arithmetically optimistic and the medium risk is understated.** Release timings show approximately 4.70 seconds for submit and 9.35 seconds for advance, about 14.06 seconds combined (`docs/spikes/bridge-persistence-step0-release/summary.json:40-42`, `docs/spikes/bridge-persistence-step0-release/summary.json:91-92`). Subtracting the plan’s 6.5–7.0-second scope saving and 4.2-second persistence saving yields roughly 2.86–3.36 seconds, not 2.4 seconds (`docs/superpowers/plans/2026-07-23-bridge-persistence-slice.md:21-28`, `docs/superpowers/plans/2026-07-23-bridge-persistence-slice.md:298-306`). Even another 0.4-second memoization saving leaves approximately 2.46–2.96 seconds. For medium, the cold allowance derived from the recorded SQLite p95 is about 697 ms, while the prior candidate validation measurement alone was about 786 ms (`docs/spikes/gate3-noninferiority-profile.md:24-27`, `docs/spikes/gate2-structural-edit-profile.md:13-15`). Task 1 should therefore be followed immediately by an actual release-mode medium residual profile and revised arithmetic before investing in the digest/protocol work.

5. **The memory guard is insufficient and is not part of the exit decision.** Comparing the last four medium samples with the first four at a 1.15 factor can miss a large stable footprint or an early spike, while big1k receives no absolute limit (`docs/superpowers/plans/2026-07-23-bridge-persistence-slice.md:246-254`). Record daemon and worker PID continuity, use a defined quiescence/GC sampling protocol, add a preregistered absolute combined-RSS cap for big1k, and feed this predicate into the amended A3 exit verdict.

6. **The exit artifact lacks required configuration provenance and regression protection.** The current runner only parses `--smoke` (`packages/live-compare/src/gate3/run-big.ts:332-339`), hardcodes its artifact name and marker (`packages/live-compare/src/gate3/report.ts:435-463`, `packages/live-compare/src/gate3/run-big.ts:417-424`), and has CI assertions tied to the prior artifact and HEAD (`packages/live-compare/src/gate3/gate3Artifact.ci.test.ts:25-29`, `packages/live-compare/src/gate3/gate3Artifact.ci.test.ts:119-151`). Add an explicit artifact-base flag, a new marker, and a corresponding CI/schema/raw-pair recomputation test. The artifact itself must record release profile, exact daemon SHA, persistent bridge mode, Task 1 and memoization toggles, and the complete command/configuration.

## Minor

1. **A1’s equivalence gate should cover real and mutated graphs, not only synthetic cases.** `children_resource` currently clones the snapshot and hashes sorted triples (`crates/strata-coordination/src/coordination/resources.rs:136-149`); the immutable generation already builds indexes suitable for this optimization (`crates/strata-coordination/src/model/graph.rs:139-181`). Extend the byte-equivalence gate (`docs/superpowers/plans/2026-07-23-bridge-persistence-slice.md:91-95`) to medium and big1k, post-`GraphGeneration::apply` states, equal `childIndex` ordering, and non-ASCII identifiers if permitted.

2. **Poison handling needs an explicit post-assertion mismatch test.** The plan says divergence poisons the worker but lists only success, failure, and crash cases (`docs/superpowers/plans/2026-07-23-bridge-persistence-slice.md:216-220`). Deliberately corrupt mirrored state, confirm the postcheck detects it, ensure the next request is refused, and verify the host kills and fully rehydrates the worker.

## Explicit answers to the nine questions

1. **A1 is a legitimate measured lever and does not require operator approval.** It is an internal complexity correction in the same coordination path, with unchanged semantics, thresholds, and benchmark boundaries. The existing implementation does unnecessary snapshot cloning and sorting (`crates/strata-coordination/src/coordination/resources.rs:136-149`), while immutable generation indexes already provide the necessary basis (`crates/strata-coordination/src/model/graph.rs:139-181`). Its equivalence gate is sufficient in principle after the real-corpus and mutation coverage described above.

2. **A2 is fair if disclosed and made symmetric at the production-policy level.** Using an optimized release binary is a production configuration correction, not threshold motion. The recorded Gate 3 artifact must remain immutable, while the new exit artifact must identify the exact release binary and SHA. “Both arms use their production-optimized implementation” is the honest symmetry rule; forcing SQLite native code to emulate an unoptimized Rust daemon would not be representative. A2 still cannot yield a valid exit decision until A3 resolves the impossible memory verdict.

3. **Use a corrected dual state digest, not a delta-chain-only digest.** The daemon should calculate the target digest from the canonical immutable graph, the worker should calculate the digest from its post-transaction SQLite state, and the host should record `(generation, digest)` only after exact equality. Generation must be transported as a decimal string, and canonical encoding must be explicitly specified and cross-language tested. This closes silent node/reference divergence within the digest’s collision assumptions. The separate savepoint fingerprint must cover non-graph SQLite tables. Exporting complete canonical bytes from Rust for every check would undermine the persistence objective, while a delta chain would not prove the applied state.

4. **Cap the delta log by both count and bytes.** A concrete initial policy is 4,096 entries or 16 MiB serialized, whichever comes first, with full hydration when the worker’s base is older than the retained floor or a batch would exceed the request frame. Concurrent sync must operate on an immutable captured batch. Boundary, truncation, and append-during-sync tests are required.

5. **Eager hydration before readiness is honest relative to the existing gate, but must be described accurately.** The daemon opens and recovers before signaling readiness (`crates/strata-coordination/src/service/server.rs:34-47`). The benchmark starts the service and resolves its target before the mutation timer (`packages/live-compare/src/gate3/kernel-child.ts:90-131`, `packages/live-compare/src/gate3/kernel-child.ts:154-193`), and the parent uses child-reported caller wall time rather than total process time (`packages/live-compare/src/gate3/runners.ts:256-269`). SQLite likewise ingests before its timed mutation section. Therefore eager hydration before readiness is consistent with the current comparison. The report should disclose service-start/hydration separately and call the sample “process-cold mutation latency,” not startup latency.

6. **Savepoints are sufficient for logical isolation in the current in-memory, single-connection design; raw physical byte identity is neither necessary nor meaningful.** The stronger requirement is a logical fingerprint across nodes, references, transactions, operations, and auxiliary state, plus integrity checks and external mirror attestation. A graph-only `(generation, digest)` assertion is insufficient.

7. **The projection does not presently support a confident under-2.5-second claim.** Direct release arithmetic gives approximately 2.86–3.36 seconds before memoization and approximately 2.46–2.96 seconds after the claimed memoization saving. Medium is a known risk because previously measured candidate validation alone exceeds the cold allowance. After Task 1, rerun release-mode big1k arithmetic and a real medium residual profile. If irreducible candidate-plus-publication work still exceeds the allowance, stop and seek a design amendment for another profiled lever rather than finishing the protocol on hope.

8. **The seven gate categories are nominally present, but the gate set is incomplete.** The plan covers protocol, synchronization, isolation, differential oracle, full suite, memory, and exit comparison. Missing cases are unpublished speculative generations, atomic G/G+1 attestation races, bounded-log truncation, explicit poison-state recovery, full-hydration digest mismatch, asymmetric frame limits, absolute big1k memory, and new-artifact CI/provenance. The sequential oracle in Task 8 (`docs/superpowers/plans/2026-07-23-bridge-persistence-slice.md:228-241`) must also include concurrent overlapping requests and publication failpoints.

9. **Task 1 should remain first, but the build order needs an earlier go/no-go checkpoint.** After Task 1 and its ablation, immediately recalculate big1k residuals and run the actual release-mode medium profile. Only then proceed to digest and protocol work. The Task 5 and Task 6 ablations are useful for attribution, but add a Task 7/savepoint ablation so isolation overhead is measured rather than absorbed into the final result.

## Overall verdict

The persistence direction is technically plausible, and A1 and A2 are acceptable amendments, but the plan is not sound enough to execute as v2 yet. It needs re-chartering because the non-negotiable unchanged Gate 3 verdict cannot produce PASS and therefore requires an explicit A3 operator amendment. That amendment should also establish the true-process memory verdict. Afterward, the plan must prevent speculative-generation hydration, make synchronization and semantic dispatch atomic, strengthen digest canonicalization and savepoint assertions, bound the delta log, and insert the early medium go/no-go checkpoint; with those changes, the slice can proceed without changing its core persistence thesis.

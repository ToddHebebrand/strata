2. **Verdict — flawed.**

   **Evidence:** A persistent worker cannot use the current candidate handler directly. Today the worker hydrates a throwaway database, calls `commit`, exports the mutated graph, and closes the database ([candidate.ts:48-68](/Users/toddhebebrand/Strata/packages/kernel-bridge/src/candidate.ts:48), [candidate.ts:86-178](/Users/toddhebebrand/Strata/packages/kernel-bridge/src/candidate.ts:86)). `commit` permanently materializes the overlay and operation log into that SQLite database ([validate.ts:237-273](/Users/toddhebebrand/Strata/packages/verify/src/validate.ts:237)). On a persistent mirror, a successfully validated but not-yet-published candidate would therefore contaminate later analyses. Validation failure, kernel invalidation, or a publication race would leave the mirror ahead of canonical redb.

   **What I’d change:** Execute each candidate inside an outer SQLite savepoint, capture its diagnostics/delta, then always roll back that savepoint before replying. Only published redb deltas may advance the persistent mirror. Assert after both successful and failed candidates that the mirror remains at the original generation and digest.

3. **Verdict — flawed.**

   **Evidence:** Probabilistic or every-K digest checking is too weak for a trusted semantic provider. Current response binding is safe because every request contains the complete snapshot; the worker merely copies request binding fields into the response ([worker.ts:250-255](/Users/toddhebebrand/Strata/packages/kernel-bridge/src/worker.ts:250), [protocol.rs:1089-1118](/Users/toddhebebrand/Strata/crates/strata-kernel/src/bridge/protocol.rs:1089)). With a stateful mirror, that echo no longer proves the worker actually analyzed the named graph. The sharpest failure is same-generation divergence: a missed or incorrectly applied reference delta causes the worker to under-report scope; Rust then derives reservations from incomplete facts, and containment cannot detect omitted resources because it checks against that same inferred scope. An ahead-of-request worker is another failure mode because delta synchronization is forward-only. Durable delta scans detect gaps, but only when explicitly used ([storage.rs:703-742](/Users/toddhebebrand/Strata/crates/strata-kernel/src/storage.rs:703)).

   **What I’d change:** Make synchronization transactional and exact: base generation/digest → contiguous canonical deltas → expected target generation/digest. The worker must compute and acknowledge its actual mirror digest before serving semantic work. A worker at generation greater than the request must never serve it; use another eligible worker or an exact-generation full-snapshot fallback. Kill all workers on service-epoch change. Sampling is acceptable for differential shadow validation, not for production mirror correctness.

   **What I’d change:** Use this minimum gate sequence:

   1. Multi-frame protocol tests: frame bounds, request correlation, deadlines, stderr bounds, crash/reap/respawn, concurrent callers, and clean EOF.
   2. Mirror tests: ordered/duplicate/gapped deltas, exact digest after synchronization, ahead-generation rejection, epoch reset, and failpoints around delta transactions.
   3. Candidate-isolation tests: successful and failing validation both leave the canonical mirror byte-identical.
   4. Deterministic differential oracle: pooled versus current one-shot full-snapshot results for semantic facts, diagnostics, delta bytes, and candidate digest across rename and add-parameter generation sequences. Use fixed sampled shadow cases; production state verification remains exact.
   5. Run the unchanged gate-1 suite.
   6. Add a true-process RSS/leak guard for daemon plus the single persistent worker; do not modify gate 3.
   7. Run the existing gate-3 harness unchanged as the sole exit gate.

## Summary

The Phase 1.5R remediation made the four-task harness **valid** and the
operator re-validation produced an **honest negative** under it: T03's
`rename_symbol` advantage replicates and is robust, but the three new
tools (`add_parameter`, `change_return_type`, `replace_body`) are not
agent-effective on real tasks. A cheap, instrumented, log-keeping round
(`decisions.md` 2026-05-15 "R3 log-based classification") then persisted
real substrate transcripts and decomposed that single negative into
**three distinct causes** — not one fundamental substrate failure.

This spec addresses **exactly two** of those three causes, with the third
explicitly deferred and documented. It is a **narrow agent-effectiveness
pass**, not a feature, not a benchmark redesign, not a tool-logic change,
not a loop redesign. Scope is fixed at three items and is not broadened:

- **P1 — system-prompt explore-then-act discipline (fixes the T05
  thrash).** On a one-line bugfix, the substrate agent made **23 tool
  calls, ZERO mutations**, opened one transaction, called `validate`
  once, and **never attempted the fix**: 14 `read_node` + 5
  `find_declarations`, looping on cheap read-only structural tools. The
  file baseline did the same task in 5 tools / 16 s. The cheap structural
  reads enable infinite stalling; the prompt does not impose a bounded
  explore→act→validate→commit loop. P1 adds a **general discipline**
  (stop exploring once the target is located; do not re-read a node
  already read; exploration finds the target, it does not re-confirm it)
  — not a task-specific script.

- **P2 — `add_parameter` tool-description legibility (fixes the T01
  hand-patching).** The agent did not trust/understand that
  `add_parameter` already updates every resolved direct callsite
  atomically in the same transaction, so it fell back to hand-patching
  callsites with `replace_body`, looped, hit an `oldText` mismatch on
  `validate`, rolled back, and re-flailed. P2 rewrites the
  `add_parameter` description in `tools.ts` so the tool's actual
  contract — declaration **and** every resolved direct callsite edited in
  one transaction; do not hand-edit callsites; higher-order/aliased uses
  are intentionally left as arity-risk sites for `validate` to surface,
  which is correct behavior, not a tool failure — is legible to the
  agent. The text stays accurate to `addParameter.ts`/`callsites.ts`
  (no overclaiming).

- **P3 — the T08 commit-gate gap is documented-not-solved.** T08's cause
  is structurally different: a **clean 15-call run** (`begin_transaction`
  → `change_return_type` → `replace_body` ×2 → `validate` (clean) →
  `commit_transaction`, terminal success) that is **confidently wrong** —
  `validate` (tsc-clean) is weaker than the task's real success criterion
  (corpus `vitest`), so the agent commits with no signal that it is
  wrong. That is a **loop/commit-gate design question**, not prompt or
  tool-description tuning. It is **out of scope** for this pass. The
  re-validation must interpret a continued T08 failure as the **known,
  documented limitation** — not a regression, not a reason to keep
  tuning — and **nothing in P1/P2 may backdoor-fix T08** (in particular,
  the prompt must **not** be told to "run the corpus tests before
  committing"; that conflates the deferred loop redesign with this scoped
  pass and contaminates the finding).

The non-negotiable invariant across this pass: **T03 must not regress.**
T03 is the proven, replicated win — the crown jewel. A prompt that breaks
T03 is worse than the prior prompt; reverting beats shipping a regression.
T03 non-regression is the hard gate (BS-P-A). If a fair, general
(non-scripted) prompt+description rework still leaves T05/T01 thrashing,
that is an honest "prompt-tuning is insufficient; the cause is deeper than
navigation/legibility" finding to report after **one** fair attempt — not
a license to iterate the prompt indefinitely (BS-P-B).

## Background

### The three-cause classification (verbatim-faithful to `decisions.md`)

The authoritative problem statement is the two newest `decisions.md`
entries. The honest negative ("Phase 1.5 re-validation: harness now
valid; T03 win replicates; new tools are NOT agent-effective") established
that Phase 1's `rename_symbol` advantage is real and replicable but
Phase 1.5's tool expansion does not generalize it: the tools pass 170
unit tests but the agent cannot effectively wield them on real tasks.
Then the instrumentation gap was fixed (`--keep-artifacts` → `logPath`
wiring), a cheap targeted keyed round (T01/T05/T08, N=1, $0.48) persisted
real transcripts, and the "R3 log-based classification" entry decomposed
the negative into **three distinct causes**:

**Cause (a) — agent navigation/decision discipline (T05 thrash).**
Transcript ground truth (persisted log
`T05-substrate-trial1-2026-05-15T21-55-04-193Z.jsonl`, read directly):
**23 tool calls, ZERO mutation calls**, 1 `begin_transaction`, 1
`validate`; **14 `read_node`** (11 consecutive) + **5
`find_declarations`** + 2 `get_references`. The agent never attempted the
fix — it loops on cheap read-only structural tools and never commits to
acting. The file baseline did the same one-line bugfix in 5 tools / 16 s.
`decisions.md` classifies this as **pure BS15-E
exploration/decision thrash** and notes it is **likely
system-prompt-tunable** (an explore-then-act budget; the cheap structural
read tools enable infinite stalling). This is **P1**.

**Cause (b) — tool legibility (T01 hand-patching).** Transcript ground
truth (persisted log
`T01-substrate-trial1-2026-05-15T21-47-28-400Z.jsonl`, read directly):
**34 tool calls**, sequence `begin_transaction` → `add_parameter` →
`replace_body` ×3 → `validate` (fail: `oldText mismatch at [52,110)`) →
`rollback_transaction` → `begin_transaction` → `replace_body` ×3 →
`add_parameter` → `validate` (fail: same `oldText mismatch`); **never a
clean `validate`**. The agent calls `add_parameter` correctly **once**,
then does not trust/understand that it already fanned the argument to
callsites, so it hand-patches the caller bodies with `replace_body`
(which races `add_parameter`'s own callsite edits and produces the
`oldText` mismatch), loops, and rolls back. `decisions.md` classifies
this as **tool-illegibility + thrash** and notes
**tool-description + prompt work, medium**. This is **P2** (the
tool-description half) plus P1's general discipline.

**Cause (c) — commit-gate weakness (T08).** Transcript ground truth
(persisted log `T08-substrate-trial1-2026-05-15T22-00-23-866Z.jsonl`,
read directly): a **clean 15-call run** — `begin_transaction` →
`change_return_type` → `replace_body` ×2 → `validate` (returns `[]`,
clean) → `commit_transaction` (`ok: true`), **terminal success**. The
agent did everything the prompt asks: it selected the right tool, made
the smallest change, validated clean, and committed. But the corpus
`vitest` fails: `validate` (tsc-clean) is **not** the task's real success
criterion, so a confidently-wrong commit passes with **no signal**.
`decisions.md` calls this the **most significant** cause and explicitly
**NOT thrash** — a **loop/design question** (the agent likely needs a
task-acceptance/test signal in-loop, not just tsc), **not a tool bug and
not prompt/description tuning**. This is **P3**, and it is **deferred**.

### Why this pass addresses only P1 and P2

`decisions.md` itself partitions the levers: causes (a) and (b) are
"likely system-prompt-tunable" / "tool-description + prompt work,
medium"; cause (c) is "a loop/design question … not a tool bug." This
pass takes exactly the (a)/(b) lever and nothing else. Conflating (c)
into this pass — e.g. by teaching the prompt to run corpus tests before
commit — would (1) be a loop/commit-gate redesign masquerading as prompt
tuning, (2) require the agent to have a filesystem/test-runner surface it
deliberately does not have (CLAUDE.md: files are not first-class; the
agent's world is the node graph; `decisions.md` hermetic-isolation entry:
the 11-tool surface, `tools: []`, `strictMcpConfig`, `settingSources: []`
are invariants), and (3) contaminate the T08 finding so a future
loop-redesign decision could no longer trust it. P3 exists to **fence
that off explicitly**.

From CLAUDE.md and `decisions.md` (hermetic-isolation entry):

- **The 11-tool surface is frozen.** No new tool, no removed tool, no

## P1 — system-prompt explore-then-act discipline

### The defect, precisely

`T05-substrate-…jsonl` (the persisted control transcript): 23 tool calls,
zero mutations. Calls 1–6 locate the target (`find_declarations` →
`read_node` → `find_declarations dateRange` → `get_references` →
`read_node` → `begin_transaction`); call 7 is a premature `validate` on
an empty transaction; calls 8–23 are **16 more reads** (`read_node` ×11
consecutive, `find_declarations` ×4, `get_references` ×1) that never lead
to a mutation. The agent **had located the target by call 6** and then
**re-explored instead of acting**. The current prompt's "Explore before
mutate" / "the goal is not to perform many tool calls" language is
descriptive, not operative: it never tells the agent **when exploration
is done** or that re-reading an already-read node is wasted motion. The
cheap, side-effect-free read tools make stalling free, so the agent
stalls.

### The exact `prompt.ts` changes

Two surgical edits to `packages/agent/src/prompt.ts`. Both are
general-discipline statements; neither names a task, symbol, tool
sequence, or count keyed to T05/T01.

**Edit 1 — append an operative "stop exploring" rule to the existing
"Explore before mutate" section.** After the section's final paragraph
("Exploration is not busywork. … you treat it as outside the
mutation."), add a new paragraph (the load-bearing addition):

> Exploration is bounded. Once you have located the declaration the task
> targets and, for a reference-sensitive change, inspected its
> references, exploration is finished — open a transaction and make the
> change. Do not re-read a node you have already read, and do not
> re-issue a query whose answer you already have; the graph does not
> change while you explore, so a second read returns the same node and
> tells you nothing new. Reading is for finding and confirming the
> target once, not for repeatedly re-confirming a target you have
> already found. If you have read the target declaration and (when
> relevant) its references and you still have not begun a transaction,
> that is the signal to begin one now, not to read more.

**Edit 2 — add one operative sentence to the existing "One worked
pattern (rename)" closing paragraph**, reinforcing the same discipline at
the place the agent reasons about the loop. After "The goal is not to
perform many tool calls; the goal is to make the correct structural
change with enough evidence that it is correct.", append:

> Repeated reads are not evidence; they are avoidance. The evidence you
> need is the located declaration, its reference set when the change is
> reference-sensitive, and a clean `validate` on the pending change —
> nothing more. When you notice you are reading rather than mutating,
> that is the moment to open a transaction and act.

### Why this is a general discipline, not a T05 script

The additions never reference a task, a symbol name, a tool-call count, a
specific tool sequence, or a benchmark identifier. They state a
domain-true invariant — *the graph is immutable during exploration, so a
re-read is informationally empty* — and a general stopping rule —
*target located (+ references when reference-sensitive) ⇒ stop exploring,
begin the transaction*. This is the same kind of worldview teaching the
Phase 3 prompt already uses (e.g. "A string literal that happens to spell
a name is not a reference") which BS-A explicitly validated as a
*reasonable worldview*, not a disguised script. It helps **any** task
where the agent would otherwise stall on cheap reads (T05 most acutely,
T01 secondarily), and it is inert on T03 (which already converges fast and
never over-reads), so it cannot regress T03 by construction.

### Token-budget impact

Edit 1 ≈ 95 words ≈ ~125 tokens. Edit 2 ≈ 55 words ≈ ~75 tokens. Total
≈ **~200 tokens added**, taking the prompt from ~2,600 to **~2,800
tokens** — well inside the design's 2000–4000-token, single-static-string,
prompt-cacheable shape (Phase 3 spec § "System prompt outline"). The
prompt remains one static string, so it stays cacheable as-is; the
cache-key changes once (the constant changed) and is then stable, exactly
as the design intends. No structural change to the prompt's section
ordering or caching story.

## P2 — `add_parameter` tool-description legibility

### The defect, precisely

`T01-substrate-…jsonl`: the agent calls `add_parameter` correctly once
(call 23, `ok: true`), then immediately `replace_body` ×3 on the caller
functions (calls 24–26) to hand-insert the argument it does not realize
`add_parameter` already inserted. `validate` then fails with
`oldText mismatch at [52,110)` — because `add_parameter` had **already
queued** the callsite text edits and the hand-patch `replace_body`
collides with them. The agent rolls back and repeats the same pattern
in a second transaction (calls 29–34). The current description says
add_parameter "insert[s] a corresponding argument at every callsite the
reference graph can resolve" and that non-direct-call references "are
left for validate to report rather than silently mis-edited" — accurate,
but it does **not** tell the agent the consequence the agent needs:
*therefore you must not hand-edit callsites yourself*. The contract is
stated; the **instruction not to fight it** is missing. Tool descriptions
are the agent's worldview (CLAUDE.md working-style); this one
under-specifies, so the agent fills the gap with the wrong instinct.

### The actual contract (from `addParameter.ts` + `callsites.ts`)

Verified against `packages/store/src/addParameter.ts` and
`packages/store/src/callsites.ts`:

- `add_parameter` queues the parameter-insertion text edit on the
  function declaration (`queueTextSpanEdit(tx, functionId, …)`), then
  calls `resolveCallsites(db, functionId)` and, **for every resolved
  direct callsite**, queues an argument-insertion text edit
  (`queueTextSpanEdit(tx, callsite.statementId, …)`) **in the same
  transaction `tx`**. The slot value is the supplied `default` or
  literally `undefined`. One `AddParameter` op is recorded over all
  affected node IDs (declaration + every edited callsite statement).
- `resolveCallsites` walks the **reference graph** (`getReferencesByTo`
  on the declaration's identifier child), skips `ImportDeclaration`
  references, and for each remaining reference determines whether the
  reference is the callee of an enclosing `CallExpression`
  (`enclosingCallWhoseCalleeIs`, transparent through parentheses). If it
  is, it is a **resolved direct callsite** and gets the argument. If it
  is **not** a direct call — passed as a higher-order value, aliased to a
  variable, or otherwise used in value position — it is recorded as a
  `nonCallReference` (an **arity-risk site**) and is **deliberately not
  edited**; `validate` surfaces the resulting arity mismatch. Unresolved
  references (missing node/statement/identifier) are likewise not edited
  and counted separately.
- Therefore: the declaration **and** every resolved direct callsite are
  changed atomically by the single `add_parameter` call. Higher-order /
  aliased uses being left for `validate` to flag is the **intended,
  correct** behavior — not a tool failure and not something the agent
  should repair by hand. Manually editing callsites after `add_parameter`
  **double-edits** them and corrupts the transaction (exactly the
  observed `oldText mismatch`).

### The exact rewritten `add_parameter` description

Replace the `addParameterTool` description string in
`packages/agent/src/tools.ts` (currently:
*"Add a parameter to a function declaration and insert a corresponding
argument at every callsite the reference graph can resolve. Requires an
open transaction; mutates the transaction overlay only. You supply the
parameter name, type, position, and optional default. Callsites are found
through the reference graph, not text search: unrelated identifiers that
merely share the name are never touched, and a reference used in a
position that is not a direct call is left for validate to report rather
than silently mis-edited."*) with, verbatim:

> Add a parameter to a function declaration AND insert the corresponding
> argument at every resolved direct callsite, all in one operation in the
> open transaction you pass. This is not a declaration-only edit: in the
> same transaction it rewrites the function's parameter list and, for
> every place the reference graph resolves as a direct call of this
> function, inserts the new argument (your `default` if given, otherwise
> `undefined`) at the matching position. You do not, and must not,
> hand-edit callsites afterward — the argument is already inserted at
> each resolved direct callsite, so editing them yourself double-edits
> the text and corrupts the transaction. Callsites are found through the
> reference graph, not text search: identifiers that merely share the
> name are never touched. References that are not direct calls —
> higher-order uses (the function passed as a value), aliased uses
> (assigned to another binding), or other value positions — are
> intentionally left unedited and reported by `validate` as arity-risk
> sites. That is correct, expected behavior, not a failure of this tool
> and not your cue to patch them with `replace_body`: inspect what
> `validate` reports and decide deliberately. Requires an open
> transaction; mutates the transaction overlay only. You supply the
> parameter name, type, zero-based position among the existing
> parameters, and an optional default expression.

This is accurate to `addParameter.ts`/`callsites.ts` with no
overclaiming: it says "resolved direct callsite" (matching
`resolveCallsites`' `enclosingCallWhoseCalleeIs` semantics), states the
`default ?? "undefined"` slot value exactly, names the three non-call
shapes `callsites.ts` actually classifies (`higher-order-value`,
`aliased-value`, `value-position`), and frames the arity-risk leave-out
as intended behavior surfaced by `validate` — which is precisely what the
code does. It adds the missing operative instruction (do **not**
hand-edit callsites; do **not** `replace_body` the arity-risk sites
reflexively) without claiming the tool does anything it does not.

### Scope discipline for P2

P2 changes **only the description string**. `addParameter.ts`,
`callsites.ts`, the Zod input schema, the handler body, and every other
tool's description are untouched. The 11-tool surface and
`STRATA_TOOL_NAMES` are unchanged. If making this description accurate
appeared to require changing `addParameter.ts`'s behavior, that would
mean the cause is not description legibility and the scoping was wrong —
BS-P-C fires (see Bail signals).

## P3 — the T08 commit-gate gap is documented-not-solved

### Statement of the deferred limitation

T08's cause is **categorically different** from P1's and P2's. It is not
navigation thrash and not tool illegibility. The persisted T08 transcript
is a *clean, correct-looking* run: the agent selected the right tool
(`change_return_type`), made the minimal supporting body edits, called
`validate`, got an empty diagnostic list, and committed — terminal
success. The defect is that **`validate` (tsc-clean) is a weaker gate
than the task's real success criterion** (the corpus `vitest` suite). The
agent committed *confidently wrong* because the substrate gave it **no
in-loop signal** that the change, while type-clean, fails the behavioral
test. `decisions.md` ("R3 log-based classification") calls this the **most
significant** of the three causes and explicitly frames it as **a
loop/design question (the agent likely needs task-acceptance/test signal
in-loop, not just tsc), not a tool bug** and not prompt/description
tuning.

### Out of scope for this pass — and the no-backdoor-fix instruction

P3 is **explicitly deferred**. This pass does **not** redesign the commit
gate, does **not** add a test/acceptance signal to the agent loop, and
does **not** alter `validate`/`commit`. Critically, **nothing in P1 or P2
may attempt to backdoor-fix T08.** In particular:

- The system prompt must **not** be given any instruction to "run the
  corpus tests before committing," "verify behavioral/test success before
  commit," or any equivalent. The agent has no test-runner or filesystem
  surface by hermetic design (CLAUDE.md; `decisions.md`
  hermetic-isolation entry), and adding such prompt language would (1) be
  a commit-gate/loop redesign disguised as prompt tuning, (2) instruct
  the agent toward a capability it deliberately does not have, and (3)
  contaminate the T08 finding so the future loop-redesign decision can no
  longer trust it as a clean, isolated observation.
- P1's discipline (stop exploring, act, validate, commit) and P2's
  description (trust `add_parameter`'s fan-out) must be written so they
  are **inert on the T08 mechanism**: they change *when/how the agent
  acts and which tool it trusts*, never *what gate must pass before
  commit*. The commit gate language in the prompt's "Verify before
  commit" / "The transaction model" sections is **unchanged** by this
  pass.

### How the re-validation must interpret a continued T08 failure

If the operator's keyed re-validation round still shows T08 committing a
tsc-clean-but-`vitest`-failing change, that outcome must be recorded and
interpreted as the **KNOWN, DOCUMENTED limitation** (this entry and the
"R3 log-based classification" `decisions.md` entry) — **it is not a
regression introduced by P1/P2, and it is not a reason to keep tuning the
prompt or the description.** T08 is included in the re-validation round
*only to confirm it is unchanged/documented*, not as a target this pass
tries to move. A T08 outcome that is *different* (better or worse)
warrants inspecting whether P1/P2 inadvertently touched the commit
mechanism — if so, that is a P1/P2 scope violation to surface (BS-P-C),
not a T08 "improvement" to claim. The legitimate next lever for T08 is a
separate, future commit-gate/in-loop-acceptance design decision, made on
the *clean* evidence this pass is careful not to contaminate.

- **No prompt section-restructure and no scripts.** P1 appends two

## Bail signals (carry as exit criteria)

A surfaced wall is more valuable than a papered-over one
(`strata-design.md` § "What success looks like"). Each is recorded
newest-first in `decisions.md` by the operator round that surfaces it.

### BS-P-A — T03 regresses (the hard gate)

If the prompt rework makes **T03** — the proven, replicated win — regress
in the re-validation (T03 substrate no longer succeeds, or its
shared-criteria/replay path no longer passes, or the byte-frozen T03
regression guards are no longer byte-identical), **STOP and surface.**
Reverting `prompt.ts`/`tools.ts` to the prior text is strictly better
than a prompt that breaks the one thing that works. T03 non-regression is
**non-negotiable**: this bail dominates all others. (P1/P2 are designed to
be inert on T03 — it converges fast, never over-reads, and does not use
`add_parameter` — so if BS-P-A fires the rework is wrong, not T03.)

### BS-P-B — prompt-tuning is insufficient (the anti-inflate analogue)

If, after **one fair, general (non-scripted)** prompt+description rework
as specified in P1/P2, T05 and/or T01 **still thrash** in the
re-validation (classified from the persisted transcripts: T05 still loops
read-only with zero mutations / no clean validate; T01 still hand-patches
callsites and rolls back), that is an **honest finding**:
*prompt-tuning is insufficient; the cause is deeper than navigation /
legibility.* **Report it. Do not iterate the prompt indefinitely.** One
fair attempt, classified from the logs, then report — exactly the
anti-inflate discipline of BS-R3 (more tuning that produces more thrash,
not success, is not progress). This is a legitimate, publishable Phase
1.5 result (the BS15-E open question), not a budget to keep spending.

### BS-P-C — scope contamination (the scoping was wrong)

If making P1 or P2 work appears to require touching the commit gate
(`validate`/`commit`/the prompt's commit-gate language) or any tool
store-logic (`addParameter.ts`/`callsites.ts`/etc.), **STOP and surface.**
That means the cause is **not** prompt/description legibility and the
three-way scoping was wrong — a finding in itself. Do not quietly
broaden the change to make a scoped item "work"; the broadening is the
signal.

## Open questions

Deliberately unresolved on paper; each carries a recommendation.

### 1. Should P1's "stop exploring" rule live in one section or be split across two?

The spec puts the operative rule in "Explore before mutate" (Edit 1) and
a reinforcing sentence in "One worked pattern (rename)" (Edit 2), on the
theory that the agent reasons about the loop in both places.
**Recommendation:** ship both as specified; they are short (~200 tokens
total) and mutually reinforcing, and the worked-pattern reinforcement is
where the agent is most likely to be deciding "read more or act now." If
the re-validation transcript shows the rule is being followed after the
first read of either section alone, a later pass can consolidate — but do
not pre-optimize on paper; let the classified transcript answer it.

### 2. Is the rewritten `add_parameter` description too long for the cached tool surface?

The rewrite roughly triples the description length. Tool descriptions are
part of the cached system context (CLAUDE.md working-style), so this adds
~150 tokens to that surface. **Recommendation:** accept the length — the
description's job is to be the agent's worldview for this tool, and the
defect was *under*-specification (the agent filled the gap with the wrong
instinct). Legibility beats brevity here; the total cached surface is
still far inside budget. If BS-P-B fires (T01 still hand-patches), the
finding is "description tuning is insufficient," not "the description was
too long" — do not iterate it down hoping brevity helps.

### 3. Does P1's discipline risk under-exploration on a genuinely ambiguous task?

The rule says "target located ⇒ stop exploring." A task where the target
is genuinely ambiguous (multiple plausible declarations) could in
principle be cut short. **Recommendation:** the rule as written already
guards this — it says stop *once you have located the declaration the
task targets* and (for reference-sensitive changes) inspected its
references; the existing "If a query returns multiple plausible
declarations, narrow by kind or inspect the returned nodes" language is
unchanged and still governs disambiguation. The discipline targets
*re-reading an already-identified target*, not *initial
disambiguation*. If a re-validation transcript shows a real
under-exploration failure on an ambiguous task, that is a distinct
finding to log, not a reason to weaken the anti-stall rule that fixes the
observed T05/T01 defect.

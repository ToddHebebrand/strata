**Note:** There is no `module_index` table. The DB has no dedicated module index table. Module nodes are stored directly in the `nodes` table with `kind='Module'` and `payload` = absolute file path. The relevant tables are `nodes` (all graph nodes), `node_references` (reference edges), `transactions`, and `operations` (operation log). Vector tables (`node_embeddings`, `commit_pattern_embeddings`) and their virtual-table shadow tables are also present.

Full schema:

```sql
CREATE TABLE nodes (
      id TEXT PRIMARY KEY,
      kind TEXT NOT NULL,
      parent_id TEXT,
      child_index INTEGER,
      payload TEXT
    );
CREATE INDEX nodes_parent_kind_idx ON nodes(parent_id, kind);
CREATE TABLE node_references (
      from_node_id TEXT NOT NULL PRIMARY KEY,
      to_node_id   TEXT NOT NULL,
      kind         TEXT NOT NULL,
      FOREIGN KEY (from_node_id) REFERENCES nodes(id),
      FOREIGN KEY (to_node_id)   REFERENCES nodes(id)
    );
CREATE INDEX node_references_to_idx ON node_references(to_node_id);
CREATE TABLE transactions (
      tx_id        TEXT PRIMARY KEY,
      started_at   INTEGER NOT NULL,
      committed_at INTEGER,
      status       TEXT NOT NULL,
      actor        TEXT NOT NULL,
      triggering_prompt TEXT);
CREATE TABLE operations (
      op_id                  TEXT PRIMARY KEY,
      tx_id                  TEXT NOT NULL,
      kind                   TEXT NOT NULL,
      params_json            TEXT NOT NULL,
      affected_node_ids_json TEXT NOT NULL,
      actor                  TEXT NOT NULL,
      ts                     INTEGER NOT NULL,
      reasoning              TEXT,
      FOREIGN KEY (tx_id) REFERENCES transactions(tx_id)
    );
CREATE INDEX operations_tx_idx ON operations(tx_id);
```

#### Module count reconciliation (1,087 DB vs 811 in Task 1)

Task 1 counted 811 `.ts` files under `library/src/` using a filter that excluded `.test.ts` and `.test-d.ts`. The ingest-batch command walked the entire `library/` directory without that filter. Breakdown:

```bash
sqlite3 /tmp/valibot.strata.db "
  SELECT 
    CASE
      WHEN payload LIKE '%.test.ts' THEN 'test (.test.ts)'
      WHEN payload LIKE '%.test-d.ts' THEN 'type-test (.test-d.ts)'
      WHEN payload NOT LIKE '%/src/%' THEN 'root-level (not under src/)'
      ELSE 'source (.ts, in src/)'
    END AS file_type,
    COUNT(*) 
  FROM nodes 
  WHERE kind='Module'
  GROUP BY file_type
  ORDER BY 2 DESC;"
```

```text
source (.ts, in src/)|566
test (.test.ts)|272
type-test (.test-d.ts)|245
root-level (not under src/)|4
```

Reconciliation:

| Category | Count |
|---|---|
| Source files in `src/` (`.ts`, non-test) | 566 |
| Test files (`.test.ts`) | 272 |
| Type-test files (`.test-d.ts`) | 245 |
| Root-level files (`mod.ts`, `playground.ts`, `tsdown.config.ts`, `vitest.config.ts`) | 4 |
| **Total ingested** | **1,087** |

Task 1's 811 = 566 source + 245 type-test files (`.test-d.ts`). The 272 `.test.ts` files + 4 root-level files = 276 "surplus" modules explain the 1,087 − 811 gap exactly. The surplus is legitimate: ingest-batch does not discriminate by file role; it ingests all `.ts` files it encounters.

**Conclusion:** DB is structurally sound. 122,256 nodes across 1,087 modules; Identifier-dominated kind distribution expected for TypeScript; estimated L1 context is ~15,700 tokens, well above the 5k floor.
